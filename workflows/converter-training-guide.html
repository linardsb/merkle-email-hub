Phase 1: Figma Conversion (Primary Training)

This is the main training method. Every conversion automatically feeds the learning loop — no manual steps needed after the conversion completes.

1

Create a Figma Connection

In the Figma Conversion tab, enter:

  • Figma File URL — the full URL from your browser (e.g. https://www.figma.com/design/ABC123/My-Email)
  • Figma Access Token — your Personal Access Token from Figma settings (starts with figd_)
  • Connection Name — a human-readable label (e.g. "Q4 Campaign Designs")

Click Create Connection. The system calls POST /api/v1/design-sync/connections and returns a Connection ID.

Alternatively, click List Existing to reuse a previously created connection.

2

Browse Frames & Select a Node

Enter the Connection ID and click Browse Frames. This fetches the Figma file structure and lists all pages and frames.

Each frame shows a Node ID like 2833:1424. Copy the Node ID of the email frame you want to convert and paste it into the "Selected Node ID" field.

Tip: Pick the top-level frame that contains the full email layout, not individual sections.

3

Write a Campaign Brief

Describe the email campaign. The Scaffolder agent uses this to make better template selection and content decisions.

Example: "Create a promotional campaign email with hero image, product showcase, CTA buttons, and branded footer. Target audience: existing customers. Tone: energetic and bold."

Choose Output Mode

ModeWhat HappensWhen to Use
Structured Converter renders HTML directly from matched components, preserving Figma structure (centering, fonts, dark mode CSS, image placement). Bypasses the Scaffolder. Confidence = 0.85. Always (recommended)
Raw HTML Falls back to the Scaffolder agent which generates from scratch using the brief + layout context. May rewrite structure and lose converter optimizations. Only if converter path fails

Run QA Gate

Yes (default) runs 14 quality checks on the output: html_validation, css_support, css_audit, dark_mode, accessibility, fallback (MSO), image_optimization, brand_compliance, and more.

4

Start Conversion & Watch Progress

Click Start Conversion. Here's what fires in sequence:

POST /api/v1/design-sync/imports          # Create import record
  ↓
POST /api/v1/design-sync/imports/{id}/convert # Trigger background task
  ↓
Background task runs:

  1. Layout Analysis (layout_analyzer.py)
     • Finds top-level frames in the Figma node tree
     • Classifies each as HEADER, HERO, CONTENT, FOOTER, etc.
     • Detects column layouts (1/2/3/4-col)
     • Extracts text blocks, images, buttons
     • VLM hybrid: if heuristic confidence < 0.9,
       merges VLM classification prediction
     → Returns list of EmailSection objects

  2. Component Matching (component_matcher.py)
     • Scores each EmailSection against 151 component
       HTML templates from email-templates/components/
     • Uses section type + content signals to pick best match
     • Extended scorer handles: countdown, testimonial,
       pricing, video, event, FAQ, zigzag layouts
     → Returns ComponentMatch per section with slug + confidence

  3. Repeating Group Detection (sibling_detector.py)
     • Groups consecutive similar sections (e.g. 3 product cards)
     • Renders them as a single container table

  4. Rendering (component_renderer.py)
     • Loads matched .html component template
     • Fills data-slot attributes with actual content
       (headings, body text, image URLs, CTA text)
     • Applies token overrides (colors, fonts, sizes)
       extracted from Figma design tokens
     • Applies background colors, dark mode classes
     → Returns rendered HTML per section

  5. Assembly
     • Wraps all sections in email shell (DOCTYPE, head,
       MSO conditionals, dark mode CSS)
     • Generates style block from design tokens
     • format_email_html() final cleanup

  6. Learning (automatic, fire-and-forget)
     • converter_traces.py → JSONL trace file
     • converter_memory.py → PostgreSQL semantic memory
     • converter_insights.py → Insight bus

  7. QA Gate (if enabled)
     • 14 checks run on the final HTML

The UI polls GET /api/v1/design-sync/imports/{id} every 3 seconds until status is completed or failed.

Even failed conversions teach the system if they got far enough to produce quality data.

Phase 2: HTML + Screenshot Upload (Reference Training)

Use this when you already have a finished HTML email and want to teach the system what "good output" looks like. This creates reference cases for regression testing.

1

Upload Training Materials

Switch to the HTML + Screenshot tab and provide:

  • HTML Email File (required) — drop or paste the production HTML email
  • Design Screenshot (optional) — PNG/JPG of the original design. Enables visual fidelity comparison.
  • Figma URL + Node ID (optional) — links to the source design for token extraction

At minimum you need the HTML file. The more inputs you provide, the richer the training case.

2

Name the Training Case

  • Case ID (slug) — e.g. starbucks-q4. Becomes the directory name in data/debug/
  • Case Name — e.g. "Starbucks Q4 Promo — hero, products, CTA, footer"
  • Source Name — where this came from: email-gallery, competitor, production
  • AI Classification — Yes (recommended) uses VLM for section classification; No uses rules only
3

Analyze, Save & Backfill

Use the action buttons in order. Save and Backfill now work end-to-end via the API — no terminal commands needed.

ButtonWhat It DoesPersists to Learning Loop?
Analyze HTML (API) Calls POST /api/v1/design-sync/import/html. Parses DOM, classifies sections (HEADER/HERO/CONTENT/FOOTER/etc.), extracts design tokens (colors, typography, spacing). Shows section count + warnings. No (read-only)
Save Training Case (API) Calls POST /api/v1/design-sync/training-cases. Creates data/debug/{case-id}/ directory with expected.html + optional design.png, registers in global manifest.yaml. All server-side. Saves files, not yet in loop
Backfill Learning Loop (API) Calls POST /api/v1/design-sync/training-cases/{id}/backfill. Parses expected.html, builds a ConversionResult, persists traces + memory + insights. For Figma-sourced cases (with structure.json), runs the full converter instead. Yes — full learning loop
Check Regression Shows and copies: make converter-regression. Compares quality metrics against baseline, flags >5% drops. CLI (clipboard)
Data Regression Shows and copies: make converter-data-regression. Runs 62 parametrized tests against all manifest-driven cases. CLI (clipboard)

End-to-End Workflow (no terminal needed)

  1. Upload HTML + optional screenshot + optional Figma URL
  2. Fill in Case ID and Case Name
  3. Click Analyze HTML to verify section detection
  4. Click Save Training Case to persist on disk
  5. Click Backfill Learning Loop to feed into memory/insights/traces

That's it. The Scaffolder will use this data on future conversions automatically.

4

How Components Are Matched & Rendered

The converter uses 151 HTML component templates stored in email-templates/components/. Here's the loading chain:

email-templates/components/ ← 151 .html files hero-block.html matched when: HERO + has image hero-text.html matched when: HERO without image editorial-2.html matched when: 2-col text + image article-card.html matched when: content card with image product-grid.html matched when: product showcase logo-header.html matched when: header with logo footer-full.html matched when: footer with links button-filled.html / button-ghost.html matched when: CTA buttons countdown-timer-*.html (4 variants) matched when: time patterns detected testimonial-*.html (3 variants) matched when: quote chars + avatar pricing-table-*.html (3 variants) matched when: currency + columns + buttons zigzag-*.html (3 variants) matched when: 3+ mixed col groups ... and 130+ more

Loading Chain

app/components/data/component_manifest.yaml   # Declares all 151 components
  ↓                                               # slug, type, slot_definitions
app/components/data/file_loader.py             # Reads manifest + loads .html files
  ↓
app/components/data/seeds.py                   # COMPONENT_SEEDS list (151 + 1 shell)
  ↓
ComponentRenderer.load()                       # Caches in memory at startup
  ↓
component_matcher.py → match_all(sections)   # Scores each section vs all components
  ↓
component_renderer.py → render_section()     # Fills data-slot attrs with real content

The Learning Effect

Each conversion makes future conversions smarter

The more diverse HTML builds you feed through the system, the richer the Scaffolder's memory becomes.

What's StoredWhereWho Reads ItEffect
Quality warnings + match confidences PostgreSQL (semantic memory) Scaffolder agent Avoids bad template picks on similar designs
Low-confidence sections (<0.6) Insight bus Scaffolder agent Gets "don't use template X for pattern Y" feedback

Training API Reference

These two endpoints power the end-to-end HTML training flow. Both require developer role authentication.

POST /api/v1/design-sync/training-cases

Save an HTML email + optional screenshot as a training case on disk and register in manifest.

FieldTypeRequiredDescription
case_idstringYesSlug (alphanumeric, hyphens, underscores, max 64 chars)
case_namestringYesHuman-readable name
html_contentstringYesFull HTML email source
source_namestringNoSource label (default: "training")
figma_urlstringNoFigma design URL
figma_nodestringNoFigma frame node ID
screenshot_b64stringNoBase64-encoded screenshot (data URL or raw)

Creates: data/debug/{case_id}/expected.html, optional design.png, updates manifest.yaml

Returns: {"case_id", "case_dir", "files_written", "manifest_updated"}

POST /api/v1/design-sync/training-cases/{case_id}/backfill

Trigger learning loop backfill for a single training case. Persists traces + memory + insights.

ParamTypeDefaultDescription
traces_onlybool (query)falseIf true, write JSONL traces only (skip DB memory + insights)

For HTML-only cases (no structure.json): parses expected.html via the HTML import adapter, builds a synthetic ConversionResult with all match confidences = 1.0.

For Figma-sourced cases (has structure.json): runs the full converter pipeline.

Returns: {"case_id", "sections_count", "traces_written", "memory_stored", "insights_count", "source"}

After feeding a batch of training data, run regression checks to verify quality hasn't dropped.

Tips for Maximum Training Value

Recommended Workflow

  1. Start with Figma tab — run 5–10 real conversions from different email designs. Prioritize variety over quantity.
  2. Switch to HTML tab — upload any production HTML emails you have as reference cases
  3. Run regression — make converter-data-regression to verify quality baseline
  4. Repeat — each batch teaches the system new patterns

What to Feed

  • Prioritize variety — feed emails that cover different component types (hero variants, product grids, CTAs, footers) rather than many similar layouts
  • Always include the Figma URL when available — enables token extraction and visual verification
  • Include screenshots — enables visual fidelity comparison (ODiff pixel diff + VLM section comparison)
  • Real production emails work best — they expose edge cases that synthetic data misses
  • Component types with lowest coverage improve quality most — check which section types have low match confidence in the Monitor tab

Troubleshooting

Common Issues

ProblemCauseFix
"Failed to fetch" on Connect Backend not running or CORS blocking file:// origin Run make dev, serve the guide via python3 -m http.server 8899, open at http://localhost:8899/
"Invalid email or password" Wrong credentials Use admin@email-hub.dev / admin (or check AUTH__DEMO_USER_PASSWORD in .env)
Conversion stuck on "converting" Background task blocked or Figma API rate limit Check backend logs. Figma allows ~30 req/min. Wait and retry.
Low confidence scores (<0.6) Section type not matching any component template well Normal — this feeds the Insight Bus. Consider adding a custom component for that pattern.
"No conversion memories found" in Monitor No conversions run yet, or memory persistence disabled Run a Figma conversion first. Check DESIGN_SYNC__CONVERSION_MEMORY_ENABLED=true.
Regression test failures after adding cases New cases may expose existing weaknesses Review failures. If quality improved overall, update baseline: python -m app.design_sync.converter_regression --update-baseline
