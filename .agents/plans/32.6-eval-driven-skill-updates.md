# Plan: Phase 32.6 — Eval-Driven Skill File Updates

## Context

L3 skill files are static — written once, updated only when someone remembers. The eval system already tracks per-agent per-criterion pass rates in `traces/analysis.json`. `failure_warnings.py` injects ephemeral prompt warnings but never fixes the underlying skill files. This pipeline closes the loop: when pass rates stay low, it generates LLM-drafted skill file patches and creates branches for human review.

### Key Insight: `amendment_suggester.py` Already Does 70% of the Work

The existing `amendment_suggester.py` already: reads `analysis.json`, filters actionable clusters (≥3 failures), loads SKILL.md, calls LLM with failure context, writes suggestion files to `traces/suggestions/`. Phase 32.6 needs to add:
1. **Pass rate threshold filtering** (currently only cluster count, not pass rate)
2. **L3 skill file targeting** (currently targets SKILL.md L2, need criterion→L3 mapping)
3. **Git branch automation** (currently writes suggestion files, no git)
4. **Tool usage analytics** (promote frequently queried facts to L2)
5. **CLI with `--dry-run` / `--agent` / `--threshold`**

### Data Flow

```
traces/analysis.json → SkillUpdateDetector.detect_update_candidates()
  → filter: pass_rate < 0.80 AND failure_count >= 5
  → map criterion → L3 skill file via CRITERION_SKILL_MAP
  → generate_patch() (LLM call, temperature=0.0)
  → apply_patches() (git branch + commit)
```

## Implementation Steps

### Step 1: Add Schemas (`schemas.py`)

Add two new dataclasses after `SkillABReport`:

```python
@dataclass(frozen=True)
class SkillUpdateCandidate:
    """A detected opportunity to update a skill file based on eval failures."""
    agent: str
    criterion: str
    pass_rate: float
    failure_count: int
    failure_cluster: str          # Pattern description
    sample_reasons: list[str] = field(default_factory=list)
    target_skill_file: str = ""   # Resolved L3 skill file path

@dataclass(frozen=True)
class SkillFilePatch:
    """A proposed addition to a skill file."""
    skill_file_path: str          # Relative path: app/ai/agents/{agent}/skills/{file}.md
    patch_content: str            # Markdown to append
    candidate: SkillUpdateCandidate
    confidence: str = "MEDIUM"    # HIGH / MEDIUM / LOW (from LLM)
```

### Step 2: Create `skill_updater.py`

Core module with `SkillUpdateDetector` class:

**`detect_update_candidates()`:**
- Load `analysis.json` (reuse `_load_analysis` pattern from `amendment_suggester.py:26-34`)
- For each `(agent, criterion)` in `pass_rates`:
  - If `pass_rate < threshold` (default 0.80) AND matching cluster has `count >= min_failures` (default 5):
    - Resolve L3 skill file via `CRITERION_SKILL_MAP`
    - Create `SkillUpdateCandidate`
- Sort by impact: `pass_rate * -1` then `failure_count * -1`

**`CRITERION_SKILL_MAP`:**
- `dict[str, dict[str, str]]` mapping `agent → criterion → L3 skill filename`
- Example: `"scaffolder": {"mso_conditionals": "mso_vml_quick_ref.md", "brief_fidelity": "email_structure.md", ...}`
- Covers all 9 agents × 5 criteria = 45 mappings
- Falls back to SKILL.md (L2) if no L3 match

**`generate_patch()`:**
- Load the target L3 skill file content (or SKILL.md fallback)
- Build LLM prompt (similar to `amendment_suggester._build_suggestion_prompt` but targeting L3 skill file, temperature=0.0)
- Call LLM, parse response for patch content + confidence
- Return `SkillFilePatch`

**`apply_patches()`:**
- Abstracted git operations via `subprocess.run`:
  - `git checkout -b skill-update/{agent}/{criterion}/{date}`
  - For each patch: append content to skill file
  - `git add {files}` → `git commit -m "fix(agents): ..."`
- Return branch name
- No force-push, no main branch modification

**Tool usage analytics integration:**
- Read tool call frequency from `traces/tool_usage.jsonl` (if exists — from 32.4)
- Aggregate by `(agent, property, client)` → count
- If count > 10: create `SkillUpdateCandidate` with `target_skill_file` pointing to SKILL.md (L2 promotion candidate)
- Flag in output as "promotion candidate" vs "patch candidate"

### Step 3: Create `scripts/eval-skill-update.py`

CLI entry point using `argparse`:

```
usage: eval-skill-update.py [--dry-run] [--threshold F] [--min-failures N] [--agent NAME]
```

- `--dry-run` (default): print candidates + proposed patches, no git
- `--threshold`: override `SKILL_UPDATE_THRESHOLD` (default 0.80)
- `--min-failures`: override `MIN_FAILURE_COUNT` (default 5)
- `--agent`: filter to one agent
- Exit code: 0 if no updates needed, 1 if patches generated
