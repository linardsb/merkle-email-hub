# Plan: 41.7 VLM-Assisted Section Type Classification (Hybrid Rule + VLM)

## Context

Rule-based `_classify_section()` in `layout_analyzer.py` fails on generic frame names ("Frame 1", non-English) and unusual layouts. Phase 41.5 added VLM fallback for **component matching** (one layer down). This phase adds VLM fallback for **section type classification** (one layer up, in `analyze_layout()`). The hybrid merge keeps rule-based fast for clear cases; VLM only overrides ambiguous ones.

### Existing Patterns

- **VLM call pattern** (vlm_classifier.py): `resolve_model_by_capabilities({VISION})` → `get_registry().get_llm()` → `provider.complete(messages)` → parse JSON → `VLMClassificationResult`
- **Cache**: bounded dict `_vlm_cache` with `_CACHE_MAX_SIZE = 512`, keyed by screenshot SHA256 hash
- **Settings mock**: `_mock_settings(enabled=True)` returns patched `get_settings` with inline `type()` objects
- **Provider mock**: `AsyncMock` with `complete.return_value = CompletionResponse(content=json_str, model="test", usage={...})`
- **`analyze_layout()` is sync**; VLM calls are async. The task description says to pass pre-computed VLM results via `vlm_classifications` param (not make it async).
- **`EmailSection.classification_confidence`** already exists (line ~132) but is never populated beyond default `None`

### Architecture Decision: Pre-computed VLM Classifications

Per task spec, `analyze_layout()` stays sync. The async VLM work happens in `converter_service.py` **before** calling `analyze_layout()`:

```
convert() → if VLM enabled:
  1. export_frame_screenshots() → dict[str, bytes]
  2. VLMSectionClassifier.classify_sections() → dict[str, VLMSectionClassification]
  3. pass vlm_classifications to analyze_layout()
```

This avoids making `analyze_layout()` async and keeps Figma API concerns out of layout analysis.

Add `VLMSectionClassifier` class:

```python
class VLMSectionClassifier:
    """Batch VLM section type classifier."""

    async def classify_sections(
        self,
        frame_screenshots: dict[str, bytes],
        frame_metadata: list[dict[str, Any]],
    ) -> dict[str, VLMSectionClassification]:
        """Classify sections via multimodal VLM. Returns node_id → classification."""
```

Implementation:
- Build multimodal message: system prompt + one `ImageBlock` per frame + text metadata per frame
- System prompt instructs: classify each into EmailSectionType + detect column layout, return JSON array
- Use `resolve_model_by_capabilities({VISION})` or config override model
- Parse JSON response → dict keyed by node_id
- Timeout via `asyncio.wait_for()` with `vlm_classification_timeout`
- On error/timeout → return empty dict, log `design_sync.vlm_classification.timeout`

section_type, rule_confidence = _classify_section(node, convention, idx, total, ...)

vlm_conf: float | None = None

if vlm_classifications and node.id in vlm_classifications:
    vlm = vlm_classifications[node.id]
    vlm_type = vlm.section_type
    vlm_conf = vlm.confidence
    threshold = get_settings().design_sync.vlm_classification_confidence_threshold

    if rule_confidence > 0.9:
        pass  # Keep rule result — high confidence
    elif section_type == EmailSectionType.UNKNOWN and vlm_conf >= threshold:
        section_type = EmailSectionType(vlm_type)
        rule_confidence = vlm_conf

        section_type = EmailSectionType(vlm_type)
        rule_confidence = vlm_conf

## Preflight Warnings

- `_classify_section()` return type change is **breaking** — every caller must handle the tuple. Search for all call sites: `analyze_layout()` line ~237, any tests calling `_classify_section` directly.
- `EmailSection` is `frozen=True` — adding fields is safe (defaults), but grep for unpacking or positional construction.
- `analyze_layout()` is called from `from_legacy()` (email_design_document.py:1139) and possibly tests — all must be updated for new `vlm_classifications` kwarg (optional, so backward-compatible).
- `convert()` is sync — do NOT add `await` inside it. Either accept pre-computed VLM results or use `asyncio.run()` / caller-side async.
