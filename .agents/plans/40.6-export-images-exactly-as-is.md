# Plan: 40.6 Export Images Exactly As-Is — No Background Color Added

## Context

When the converter pipeline exports images from Figma, it must produce pixel-perfect copies of
what Figma renders — including frame backgrounds set by designers. The pipeline must never add
`background-color` CSS to image containers in generated HTML. Observed on Mammut Eiger Nordwand:
raw image fills had transparent backgrounds, causing visual gaps in product grids.

## Research Summary

### Image Export Flow (current)

```
layout_analyzer._walk_for_images()  →  ImagePlaceholder list
  ↓
import_service._collect_image_node_ids()  →  flat list[str] of node_ids
  ↓
service.export_images() / download_assets()  →  Figma API GET /v1/images/{file_ref}?ids=...
  ↓
import_service._build_design_context()  →  { "image_urls": {node_id: url} }
  ↓
converter.node_to_email_html()  →  <img src="" data-node-id="...">
  ↓
import_service._fill_image_urls()  →  replaces src="" with real URL
```

### Core Problem

In `_walk_for_images()` line 866-880: when a FRAME wraps a single IMAGE child, the code records
the **child IMAGE's** `node_id` (not the frame's). The Figma export API then renders just the
image fill — transparent PNG without the frame's background fills (e.g., `#F0F0F0` gray).

**Fix:** For FRAME-wrapping-single-IMAGE, export the FRAME node (which includes background fills),
but keep the IMAGE `node_id` for HTML `data-node-id` (URL mapping still works via the export
returning the requested node_id).

### Background-Color in Converter Output

Current state is **mostly correct**:
- `converter.py:756-771` — IMAGE nodes: no `background-color` ✓
- `converter.py:782-800` — FRAME with `image_ref` (no children): no `background-color` ✓
- `converter.py:828-834` — FRAME with `fill_color`: adds `bgcolor` on `<table>` ← this is for
  structural sections (text backgrounds), NOT image containers. Correct behavior.
- `converter.py:609` — Button `<td>` bg-color. Not image-related. ✓

**No code currently adds `background-color` to image `<td>` cells.** The main fix is on the
export side (targeting frame nodes), not on the HTML generation side.

```python
def _check_no_bgcolor_on_image_containers(html: str) -> list[str]:
    """Flag any background-color on elements containing <img>."""
    warnings: list[str] = []
    # Parse HTML and check <td> elements containing <img> for background-color
    # Use regex for lightweight check (full parser overkill for this)
    import re
    pattern = re.compile(
        r'<td[^>]*(?:background-color|bgcolor)[^>]*>[^<]*<img',
        re.IGNORECASE | re.DOTALL,
    )
    for match in pattern.finditer(html):
        warnings.append(f"Image container has background-color: {match.group()[:80]}")
    return warnings
```

### Step 7: Add dimension validation

**File:** `app/design_sync/figma/layout_analyzer.py` or new utility

Add a validation step comparing exported image dimensions against `absoluteBoundingBox`:

```python
def validate_image_dimensions(
    placeholder: ImagePlaceholder,
    exported_width: int,
    exported_height: int,
    scale: float = 2.0,
) -> str | None:
    """Return warning if exported dims don't match expected (within 1px tolerance)."""
    if placeholder.width is None or placeholder.height is None:
        return None
    expected_w = int(placeholder.width * scale)
    expected_h = int(placeholder.height * scale)
    if abs(exported_width - expected_w) > 1 or abs(exported_height - expected_h) > 1:
        return (
            f"Image {placeholder.node_id} dimension mismatch: "
            f"exported {exported_width}×{exported_height}, "
            f"expected {expected_w}×{expected_h} (@{scale}x)"
        )
    return None
```
