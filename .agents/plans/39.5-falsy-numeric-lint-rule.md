# Plan: 39.5 Custom Lint Rules for Pipeline Anti-Patterns

## Context

`0.0 or 1.0` evaluates to `1.0` in Python because `0.0` is falsy. This caused bugs in opacity, padding, font size, and positioning throughout Phase 38. A static analysis rule prevents reintroduction.

- No `.semgrep/` directory for custom rules — **needs creation**

## Implementation Strategy

Semgrep custom rules are the primary approach. A pre-commit regex grep serves as a fast secondary check. Both wire into CI.

### Why Semgrep over just regex

Semgrep understands Python AST — it can match `$X or $NUMERIC` structurally, reducing false positives on string/list/dict `or` usage. The regex fallback catches cases Semgrep's pattern might miss (e.g., multi-line expressions).

## Implementation Steps

### Step 1: Create `.semgrep/rules/falsy-numeric-trap.yaml`

```yaml
rules:
  - id: falsy-numeric-or-default
    patterns:
      - pattern: $X or $DEFAULT
      - metavariable-regex:
          metavariable: $DEFAULT
          regex: '^-?[0-9]+(\.[0-9]+)?$'
    paths:
      include:
        - app/design_sync/
    message: >-
      Falsy-numeric trap: `$X or $DEFAULT` returns $DEFAULT when $X is 0 or 0.0.
      Use `$X if $X is not None else $DEFAULT` instead.
    languages: [python]
    severity: ERROR
    metadata:
      category: correctness
      technology: [python]
      references:
        - https://docs.python.org/3/library/stdtypes.html#truth-value-testing
```

**Key design decisions:**
- `metavariable-regex` on `$DEFAULT` restricts to numeric literals → `name or "default"` won't match
- Scoped to `app/design_sync/` via `paths.include` — avoids noise in other modules
- Severity `ERROR` — CI-blocking

### Step 2: Create Semgrep test file `.semgrep/tests/falsy-numeric-trap.py`

Standard Semgrep test format with `# ruleid:` and `# ok:` annotations:

```python
# ruleid: falsy-numeric-or-default
x = opacity or 1.0

# ruleid: falsy-numeric-or-default
y = padding_top or 0

# ruleid: falsy-numeric-or-default

# ruleid: falsy-numeric-or-default

# ok: falsy-numeric-or-default
name = value or "default"

# ok: falsy-numeric-or-default
items = data or []

# ok: falsy-numeric-or-default
config = settings or {}

# ok: falsy-numeric-or-default
x = value if value is not None else 0.0
```

Symlink or copy of the rule file — Semgrep test runner expects rule + test in same dir. Alternatively, run `semgrep --test --config .semgrep/rules/ .semgrep/tests/`.

### Step 5: Add pre-commit regex fallback

Add to `.pre-commit-config.yaml` after the Bandit hook:

```yaml
  - id: falsy-numeric-trap
    name: falsy numeric or-default check
    entry: bash -c 'grep -rn "\\bor [0-9]" app/design_sync/ --include="*.py" | grep -v "# noqa: falsy-or" | grep -v "test_" && echo "ERROR: Use \"x if x is not None else default\" instead of \"x or default\"" && exit 1 || exit 0'
    language: system
    pass_filenames: false
    files: ^app/design_sync/.*\.py$
```

## False Positive Analysis

| Expression | Matches? | Correct? |
|------------|----------|----------|
| `opacity or 1.0` | Yes | True positive — 0.0 is valid opacity |
| `name or "default"` | No | Correct — string default, not numeric |
| `items or []` | No | Correct — list default |
| `height or 0` | Yes | True positive — 0 is valid dimension |
| `x if x is not None else 0` | No | Correct — proper pattern |
| `count or 1` | Yes | True positive — 0 is valid count |
| `len(items) or 1` | Yes | **Borderline** — `len()` returns int, but `len() == 0` means empty. May want `# noqa: falsy-or` |

The `metavariable-regex` on `$DEFAULT` matching only numeric literals keeps false positives near zero. The only edge case is intentional "treat zero as missing" semantics (rare, documentable with `# noqa`).
