# Plan: 45.3 — Scheduled Ontology Sync & Rendering Baseline Regeneration

## Context

The CanIEmail ontology drifts as new email client data is published. Rendering baselines go stale as emulators update. Both currently require manual `make` targets that are easy to forget. This task registers two cron jobs using the existing scheduling infrastructure to automate both processes.

## Implementation Steps

### Step 1: `app/scheduling/jobs/ontology_sync.py` (~70 lines)

Follow `qa_sweep.py` pattern exactly. Structure:

```python
"""Scheduled CanIEmail ontology sync.

Weekly sync of CSS property support data from the Can I Email repository.
Wraps CanIEmailSyncService and stores run results in Redis for the
notification layer (45.5).
"""
from __future__ import annotations

import json
from datetime import UTC, datetime
from typing import Any

from redis.asyncio import Redis

from app.core.config import get_settings
from app.core.logging import get_logger
from app.core.redis import get_redis
from app.knowledge.ontology.sync.service import CanIEmailSyncService
from app.scheduling.registry import scheduled_job

logger = get_logger(__name__)

_KEY_PREFIX = "scheduling:ontology_sync"
_LATEST_KEY = f"{_KEY_PREFIX}:latest"
_RESULT_TTL_SECONDS = 30 * 86400  # 30 days
```

`@scheduled_job(cron="0 3 * * 0")` — Sunday 03:00 UTC.

Job logic:
1. Check `settings.ontology_sync.enabled` — return early if disabled
2. `started_at = datetime.now(UTC)`
3. Create `CanIEmailSyncService()`, call `await service.sync(dry_run=settings.ontology_sync.dry_run)`
4. Build result dict from `SyncReport` fields: `new_properties`, `updated_levels`, `new_clients`, `errors`, `commit_sha`, `dry_run`, `duration_ms`
5. Store via `_store_result()` helper (same pattern as qa_sweep: dated key + latest pointer + TTL)
6. Log `ontology_sync.completed` or `ontology_sync.failed` with structured context

`_build_result()` and `_store_result()` private helpers — mirror qa_sweep exactly.

### Step 2: `app/scheduling/jobs/rendering_baselines.py` (~75 lines)

Same pattern. `@scheduled_job(cron="0 4 1,15 * *")` — 1st and 15th at 04:00 UTC.

```python
"""Scheduled rendering baseline regeneration.

Biweekly regeneration of visual regression baselines using the golden
template library and all configured email client profiles.
"""
```

Job logic:
1. Check `settings.rendering.provider != "mock"` — skip if no real rendering configured
2. `started_at = datetime.now(UTC)`
3. Create `BaselineGenerator()`, call `await generator.generate_baselines()`
4. Build result from `BaselineManifest`: `baseline_count`, `template_slugs`, `profile_ids`, `emulator_versions`, `duration_ms`
5. Store via `_store_result()` (same pattern)
6. Log `rendering_baselines.completed` or `rendering_baselines.failed`

## Preflight Warnings

- `BaselineGenerator.generate_baselines()` does filesystem I/O (writes PNGs). In tests, mock the entire `BaselineGenerator` class, not individual methods.
- `CanIEmailSyncService.sync()` makes HTTP calls to GitHub. In tests, mock the service class.
- Both jobs share the Redis `_store_result` pattern from qa_sweep — keep key prefixes unique (`scheduling:ontology_sync:*`, `scheduling:rendering_baselines:*`).
