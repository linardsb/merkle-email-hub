# Plan: 46.2 LLM Provider Key Rotation

## Context

Batch eval runs (`make eval-full`) make hundreds of LLM calls in rapid succession. A single Anthropic key with a 60 RPM limit throttles the entire run. Rotating across N keys gives N× throughput. Wire `CredentialPool` (from 46.1) into provider adapters so every `complete()`/`stream()` call acquires a rotated credential.

### Architecture

```
ChatService.chat() → resolve_model(tier) → "claude-opus-4-20250514"
  → registry.get_llm("anthropic") → AnthropicProvider
  → provider.complete(messages, model_override=model)
    └── NOW: single self._client created at __init__ with settings.ai.api_key
    └── NEW: if pool enabled → lease = await pool.get_key() → use lease.key
```

`call_with_fallback()` iterates `FallbackChain.entries`, calling `provider.complete()` on each. Key rotation is **transparent to the fallback layer** — each provider internally rotates keys. If all keys for a provider are exhausted (`NoHealthyCredentialsError`), it surfaces as a retryable error (503) and triggers fallback to the next provider:model in the chain.

### CredentialPool API (from 46.1)

```python
pool = get_credential_pool("anthropic")      # singleton per service
lease = await pool.get_key()                  # round-robin, skips cooled-down keys
# lease.key = "sk-ant-abc123..."
# lease.key_hash = "a1b2c3d4e5f6"
await lease.report_success()                  # resets failure count
await lease.report_failure(status_code=429)   # enters cooldown + backoff
# Raises NoHealthyCredentialsError when all keys exhausted
```

## Preflight Warnings

- `AnthropicProvider.__init__()` creates `self._client` at init time. With pool enabled, we need to handle the case where `self._client` is `None` — use `_get_client()` helper instead.
- `OpenAICompatProvider.__init__()` builds headers dict with API key at init. Per-request header override via `httpx.AsyncClient.post(headers=...)` merges with base headers — verify this works.
- `_is_retryable()` currently treats 401/403 as **non-retryable**. `NoHealthyCredentialsError` (503 subclass) must be handled as a separate case before the status-code check — it's retryable at the **provider level** (try different provider), not at the **key level** (don't retry same exhausted pool).
- **CRITICAL: MagicMock settings in 17 existing test blocks.** `test_phase22_integration.py` (14 blocks) and `test_cost_governor.py` (3 blocks) instantiate providers with `MagicMock` settings. After 46.2, `settings.credentials.enabled` returns a truthy MagicMock → enters pool branch → calls `get_credential_pool()` → `NoHealthyCredentialsError`. **Fix**: Guard pool init with `isinstance(settings.credentials.pools, dict)` check — MagicMock fails this, preventing false activation. This avoids touching 17 test blocks.

## Security Checklist

- No new endpoints — adapter-internal changes only
- API keys never logged (only `key_hash` prefix in structured logs)
- `CredentialLease.key` stays in memory only for the duration of the call
- `NoHealthyCredentialsError` message does not leak key values
- Redis state stores `key_hash`, not raw keys
