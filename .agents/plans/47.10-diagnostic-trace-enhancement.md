# Plan: 47.10 Diagnostic Trace Enhancement

## Context

Developers need visibility into which sections used VLM classification, verification corrections, or custom generation â€” for debugging and tuning thresholds. Currently `SectionTrace` captures only basic matching/rendering data; verification loop results and custom generation metadata are logged but not persisted in the diagnostic report.

## Research Summary

**Core files:**
- `app/design_sync/diagnose/models.py` â€” `SectionTrace` (frozen dataclass, 12 fields) + `DiagnosticReport` (17 fields)
- `app/design_sync/diagnose/runner.py` â€” `DiagnosticRunner.run_from_structure()` 6-stage pipeline, calls `build_section_traces()` at end
- `app/design_sync/visual_verify.py` â€” `VerificationLoopResult` has `iterations`, `final_fidelity`, `total_corrections_applied`, `converged`
- `app/design_sync/custom_component_generator.py` â€” returns plain `str` (HTML); slug `"custom-generated"` set in converter_service
- `app/design_sync/converter_service.py` â€” `ConversionResult` already has `verification_iterations`, `verification_initial_fidelity`, `verification_final_fidelity` fields (set in `_apply_verification()`)

**Key insight:** DiagnosticRunner runs conversion only (no verification). Verification metadata must be passed in as optional params or plumbed from `ConversionResult`.

**Existing patterns:**
- Logger: `from app.core.logging import get_logger; logger = get_logger(__name__)`
- Events: `"design_sync.verify_loop.iteration"`, `"diagnose.rendering_failed"` etc.
- `SectionTrace` is frozen dataclass with default values on optional fields


Add 2 new fields to `DiagnosticReport`:

```python

```

### Step 2: Update build_section_traces() in runner.py

Add optional parameters to `build_section_traces()`:

```python
def build_section_traces(
    layout: DesignLayoutDescription,
    matches: list[ComponentMatch],
    rendered: list[RenderedSection],
    *,
    verification_results: dict[int, tuple[float, int]] | None = None,  # section_idx â†’ (fidelity, corrections_count)
    generation_methods: dict[int, str] | None = None,  # section_idx â†’ method
    vlm_classifications: dict[int, tuple[str, float]] | None = None,  # section_idx â†’ (type, confidence)
) -> list[SectionTrace]:
```

In the trace-building loop, populate new fields from these dicts (defaulting when key absent).

### Step 3: Update run_from_structure() signature and report building

Add optional kwargs to `run_from_structure()`:

```python
def run_from_structure(
    self,
    structure: DesignFileStructure,
    tokens: ExtractedTokens,
    *,
    raw_figma_json: dict[str, Any] | None = None,
    target_clients: list[str] | None = None,
    verification_result: VerificationLoopResult | None = None,
    generation_methods: dict[int, str] | None = None,
    vlm_classifications: dict[int, tuple[str, float]] | None = None,
) -> DiagnosticReport:
```

- Pass `verification_result`, `generation_methods`, `vlm_classifications` through to `build_section_traces()`
- Extract per-section fidelity from `verification_result.iterations[-1].section_scores` if available
- Extract per-section corrections count from iteration corrections filtered by `section_idx`
- Set `DiagnosticReport.verification_loop_iterations` and `final_fidelity` from `verification_result`

### Step 4: Add structured logging events

In `runner.py`, after building section traces with verification data:

| Event | When | Key Fields |
|-------|------|------------|
| `diagnose.verification_metadata_attached` | verification_result is not None | iterations, final_fidelity, sections_with_corrections |
| `diagnose.custom_generation_traced` | generation_methods has entries | count, section_indices |
| `diagnose.vlm_classification_traced` | vlm_classifications has entries | count, section_indices |

Use `logger.info()` for the verification event, `logger.debug()` for custom gen and VLM classification.
