# Plan: 32.4 Agent Knowledge Lookup Tool

## Context

Create `lookup_client_support` + `lookup_client_support_batch` tools callable by all agents during LLM execution, backed by the centralized client matrix (32.1). Agents query specific facts at decision time instead of pre-loading full L3 reference files. Lightweight, deterministic alternative to the Knowledge agent's RAG pipeline.

### 2. `app/ai/agents/tools/client_lookup.py`
Core implementation — two tool classes.

```python
# ~150 lines
```

**`ClientLookupParams`** (BaseModel):
- `query_type: Literal["css_support", "dark_mode", "known_bugs", "size_limits", "font_support"]`
- `client_id: str`
- `property: str | None = None` — CSS property for css_support/font_support queries

**`ClientLookupResult`** (BaseModel):
- `client: str`
- `query_type: str`
- `result: dict[str, Any]`
- `workaround: str | None = None`
- `confidence: float = 1.0`

**`ClientLookupTool`** (implements `ToolProvider`):
- `name = "lookup_client_support"`
- `description` = tool description for LLM
- `parameters` property → JSON Schema dict from `ClientLookupParams`
- `async execute(**params) -> str`:
  1. Parse params into `ClientLookupParams`
  2. `matrix = load_client_matrix()`
  3. `profile = matrix.get_client(params.client_id)`
  4. If not found → return JSON `{"error": "Unknown client", "available_clients": [c.id for c in matrix.clients]}`
  5. Switch on `query_type`:
     - `css_support` → `matrix.get_css_support(client_id, property)` → serialize `CSSSupport`
     - `dark_mode` → `matrix.get_dark_mode(client_id)` → serialize `DarkModeProfile`
     - `known_bugs` → `matrix.get_known_bugs(client_id)` → serialize list
     - `size_limits` → `profile.size_limits` → serialize `SizeLimits`
     - `font_support` → `profile.supported_fonts` → serialize
  6. Build `ClientLookupResult`, return `result.model_dump_json()`
  7. Log: `logger.info("agents.client_lookup.query", agent_name=..., query_type=..., client_id=..., property=...)`

**`MultiClientLookupTool`** (implements `ToolProvider`):
- `name = "lookup_client_support_batch"`
- `description` = batch query description
- `async execute(**params) -> str`:
  1. Accept `client_ids: list[str]`, `properties: list[str]`, `query_type: str`
  2. For each client × property → call single lookup logic
  3. Return JSON matrix: `{"results": [{"client": ..., "property": ..., "support": ..., "workaround": ...}, ...]}`
  4. Log: `logger.info("agents.client_lookup.batch_query", client_count=len(client_ids), property_count=len(properties))`

**Helper:** `_get_tool_definitions() -> list[ToolDefinition]` — returns JSON Schema definitions for both tools (for LLM function calling).

- `load_client_matrix` uses `@lru_cache` singleton — tests MUST call `.cache_clear()` in fixtures to avoid cross-test pollution (pattern already established in `test_client_matrix.py`)
- Blueprint engine `_build_node_context()` is a large method (~250 lines) — add tool injection at the end, after existing metadata blocks, to minimize merge conflict risk
- `ToolProvider.execute()` returns `str`, not structured data — serialize results to JSON string
- No new API endpoints → no auth/rate-limiting needed
- Tool is read-only from static YAML → no security concerns beyond what ClientMatrix already handles
