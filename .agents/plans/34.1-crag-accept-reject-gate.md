# Plan: 34.1 Accept/Reject Gate on CRAG Corrections

## Context

The CRAG validation loop (`validation_loop.py`) currently accepts any LLM correction longer than 50 chars. This lets through corrections that introduce new unsupported CSS, break MSO conditionals, drop sections, or inflate HTML. The fix: re-scan corrected output with `unsupported_css_in_html()` and only accept if qualifying issue count strictly decreased.

## Implementation Steps

### Step 1: Add accept/reject gate in `validation_loop.py`

After the existing length gate (line 129-131), before the corrections list construction (line 133), insert:

```python
# Accept/reject gate: verify correction actually reduced qualifying issues
post_issues = unsupported_css_in_html(corrected)
post_qualifying = [
    issue for issue in post_issues
    if severity_order.get(str(issue["severity"]), 2) <= threshold
]

if len(post_qualifying) >= len(qualifying):
    logger.warning(
        "agents.crag.correction_rejected",
        pre_issues=len(qualifying),
        post_issues=len(post_qualifying),
        reason="correction did not reduce qualifying issues",
    )
    return html, []
```

Key points:
- `unsupported_css_in_html` is already imported (line 5) — zero new dependencies
- `severity_order`, `threshold`, `qualifying` are already in scope from lines 46-50
- Keeps existing `len(corrected) < 50` gate above as a fast short-circuit
- Returns `(html, [])` on rejection — same failure-safe contract as all other exit paths
- Pure regex scan, near-zero latency

### Step 2: Update existing test for happy-path correction

In `test_qualifying_issue_triggers_llm_correction`: the mock for `unsupported_css_in_html` currently returns the same issues for any input. After the gate, it will be called twice (once for original, once for corrected). Update the mock to return issues on first call and empty list on second call:

```python
patch(
    "app.ai.agents.validation_loop.unsupported_css_in_html",
    side_effect=[issues, []],  # pre: issues, post: none
)
```

## Preflight Warnings

- The existing `test_qualifying_issue_triggers_llm_correction` mocks `unsupported_css_in_html` as a simple return_value. After this change it gets called twice, so the mock must use `side_effect` instead. Update this test (Step 2) or it will false-pass (the gate would accept because second call returns same issues as first, but `qualifying` list was built from the first call's result which is the same object).
- Similarly check `test_no_fallback_still_instructs_removal` and `test_warning_severity_threshold` — both mock `unsupported_css_in_html` and may need `side_effect` if the LLM mock returns valid HTML (>50 chars).
