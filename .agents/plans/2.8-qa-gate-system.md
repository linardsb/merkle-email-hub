# Plan: 2.8 QA Gate System — Backend Completion

## Context

The QA engine backend has solid foundations: 10 check implementations, database models (`QAResult`, `QACheck`), schemas, a service orchestrator, and a single `POST /api/v1/qa/run` endpoint. However, it's missing several pieces needed to support the full UI flow described in TODO.md task 2.8:

1. **No repository layer** — service writes directly to DB (violates VSA pattern)
2. **No GET endpoints** — can't retrieve past results, no history view
3. **No override flow** — no model/endpoint for admin override with justification
4. **No template version linkage** — results aren't linked to template versions for audit
5. **No tests** — `app/qa_engine/tests/` is empty

```python
class QAEngineService:
    def __init__(self, db: AsyncSession) -> None:
        self.db = db
        self.repository = QARepository(db)

    # Existing — refactored to use repository
    async def run_qa(self, data: QARunRequest) -> QAResultResponse: ...

    # New methods
    async def get_result(self, result_id: int) -> QAResultResponse:
        """Get a single QA result by ID, including checks and any override."""
        result = await self.repository.get_result_with_checks(result_id)
        if not result:
            raise QAResultNotFoundError(f"QA result {result_id} not found")
        override = await self.repository.get_override_by_result_id(result_id)
        return self._to_response(result, override)

    async def list_results(
```

```python
        build_id=build_id, template_version_id=template_version_id,
    )

# NEW: Override failing checks (developer+ only)
@router.post("/results/{result_id}/override", response_model=QAOverrideResponse, status_code=status.HTTP_201_CREATED)
@limiter.limit("10/minute")
async def override_qa_result(
    request: Request,
    result_id: int,
    data: QAOverrideRequest,
    service: QAEngineService = Depends(get_service),  # noqa: B008
    current_user: User = Depends(require_role("developer")),  # noqa: B008

    _ = request
    return await service.override_result(result_id, data, current_user)
```

**IMPORTANT — Route ordering:** The `/results/latest` GET must be defined BEFORE `/results/{result_id}` GET in the router, otherwise FastAPI will try to parse "latest" as an integer `result_id` and return 422. Order:
1. `GET /results/latest`
2. `GET /results/{result_id}`
3. `GET /results`
4. `POST /results/{result_id}/override`

## Key Decisions

1. **`template_version_id` is nullable** — QA can run against raw HTML without a saved template version (e.g., during editing)
2. **`build_id` is also nullable** — QA can run against HTML that hasn't been through the Maizzle build pipeline
3. **Override validates check names** — only actually-failed checks can be overridden (prevents overriding passing checks)
4. **One override per result** — subsequent overrides replace the previous one (simple model, full audit via updated_at)
5. **Route ordering matters** — `/results/latest` must precede `/results/{result_id}` to avoid FastAPI path conflict
6. **Brand compliance stays as placeholder** — it's functional (always passes), configurable per-client is a Phase 3+ concern
