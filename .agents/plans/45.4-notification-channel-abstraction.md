# Plan: 45.4 Notification Channel Abstraction (Slack, Teams, Email)

## Context

Workflow events (build failures, QA regressions, approval requests, rendering issues) are invisible unless someone is actively using the CMS. This adds a pluggable notification system with a channel abstraction layer so teams get push-style alerting via Slack, Teams, or email.

### Step 2: Notification model (`app/notifications/channels.py`)

```python
from __future__ import annotations

from dataclasses import dataclass, field
from typing import Any, Protocol, runtime_checkable

type Severity = str  # "info" | "warning" | "error"


@dataclass(frozen=True, slots=True)
class Notification:
    event: str              # e.g. "qa.regression_detected"
    severity: Severity      # "info", "warning", "error"
    title: str
    body: str
    project_id: int | None = None
    metadata: dict[str, Any] = field(default_factory=dict)


@dataclass(frozen=True, slots=True)
class NotificationResult:
    channel: str            # "slack", "teams", "email"
    success: bool
    error: str | None = None


@runtime_checkable
class NotificationChannel(Protocol):
    """Protocol for notification channel implementations."""

    @property
    def name(self) -> str: ...

    async def send(self, notification: Notification) -> NotificationResult: ...
```

### Step 3: Slack channel (`app/notifications/slack.py`)

- Class `SlackChannel` implementing `NotificationChannel`
- Uses `httpx.AsyncClient` + `resilient_request` for POST to webhook URL
- Formats notification as Slack Block Kit: header block (title), section block (body), context block (event + severity + timestamp)
- Severity → emoji mapping: info=:information_source:, warning=:warning:, error=:rotating_light:
- Returns `NotificationResult(channel="slack", success=True/False, error=...)`
- Catches `httpx.HTTPError` and returns failure result (no raise)

### Step 4: Teams channel (`app/notifications/teams.py`)

- Class `TeamsChannel` implementing `NotificationChannel`
- Uses `httpx.AsyncClient` + `resilient_request` for POST to webhook URL
- Formats as Teams Adaptive Card: `AdaptiveCard` schema with TextBlock title + body + FactSet metadata
- Severity → color mapping: info=good, warning=attention, error=default (with accent color)
- Same error handling pattern as Slack

### Step 5: Email channel (`app/notifications/email_channel.py`)

- Class `EmailChannel` implementing `NotificationChannel`
- Uses `aiosmtplib.send()` following the pattern in `app/rendering/sandbox/smtp.py`
- Builds `email.message.EmailMessage` with plain text body + HTML alternative
- Subject line: `[{severity.upper()}] {title}`
- Sends to all configured `email_to_addrs`
- Catches `aiosmtplib.SMTPException` and returns failure result

### Step 6: Router (`app/notifications/router.py`)

```python
class NotificationRouter:
    """Routes notifications to configured channels."""

    def __init__(self, channels: list[NotificationChannel]) -> None:
        self._channels = channels

    @classmethod
    def from_settings(cls, config: NotificationsConfig) -> NotificationRouter:
        """Build router from app settings, instantiating enabled channels."""
        channels: list[NotificationChannel] = []
        if config.slack_enabled and config.slack_webhook_url:
            channels.append(SlackChannel(url=config.slack_webhook_url, timeout=config.slack_timeout))
        if config.teams_enabled and config.teams_webhook_url:
            channels.append(TeamsChannel(url=config.teams_webhook_url, timeout=config.teams_timeout))
        if config.email_enabled and config.email_smtp_host:
            channels.append(EmailChannel(
                smtp_host=config.email_smtp_host,
                smtp_port=config.email_smtp_port,
                from_addr=config.email_from_addr,
                to_addrs=config.email_to_addrs,
            ))
        return cls(channels)

    async def notify(self, notification: Notification) -> list[NotificationResult]:
        """Send notification to all configured channels. Fire-and-forget with logging."""
        results: list[NotificationResult] = []
        for channel in self._channels:
            try:
                result = await channel.send(notification)
            except Exception:
                logger.exception("notifications.channel_send_failed", extra={"channel": channel.name})
                result = NotificationResult(channel=channel.name, success=False, error="unexpected error")
            results.append(result)
            if not result.success:
                logger.warning("notifications.delivery_failed", extra={
                    "channel": result.channel, "event": notification.event, "error": result.error,
                })
        return results
```

## Preflight Warnings

- `app/core/config.py` is a large file with many config classes — add `NotificationsConfig` in alphabetical position near `SchedulingConfig` to keep order
- `get_settings` uses `@lru_cache` — tests must call `get_settings.cache_clear()` when patching env vars
- The `resilient_request` function is in `app/connectors/http_resilience.py` and is tightly coupled to ESP error handling (`ESPConflictError` on 409). For notification webhooks, 409 is not an ESP conflict — we should use `httpx.AsyncClient` directly with a simple retry rather than `resilient_request`. This avoids false `ESPConflictError` raises.

## Security Checklist

No new API endpoints in this phase (notification sending is internal service-layer only). Security considerations:

- [ ] Webhook URLs stored in env vars or Project JSON column — never logged (PII redaction covers URLs)
- [ ] SMTP credentials in env vars only — never in database
- [ ] Notification body must not include raw user input without sanitization (use structured templates)
- [ ] `email_to_addrs` validated as email format in config
- [ ] Timeout on all HTTP calls (30s default) to prevent hanging
- [ ] No secrets in notification payloads (metadata dict should not contain API keys)
