## 1 · Scope & non-negotiables

### 🔒 VISUAL IDENTITY LOCK — applies to every phase, every component, every page

> **Colors, fonts, icons, and corner radii are IMMUTABLE across all 8 phases.** Every sub-plan, every component, every page in this initiative MUST use the existing system. Zero exceptions, no "for this one card" overrides, no "just for the demo" hex codes. This rule is enforced by ESLint and verified in every phase's gate.
>
> **What stays exactly as today:**
> - **Colors** — the 3-tier OKLCH token system in `cms/packages/ui/src/tokens.css` (brand red `#E4002B`, brand teal `#06757E`, navy sidebar `#12295D`, purple-gray neutrals at hue 270, danger orange-red at hue 40). Components use **semantic tokens only** (e.g. `var(--color-foreground)`, `var(--color-interactive)`) — never hex values, never primitive Tailwind colors like `text-blue-500`.
> - **Fonts** — `--font-sans: "Proxima Nova"` (primary, Adobe Fonts) → "Space Grotesk" (fallback) → system. `--font-mono: "JetBrains Mono"`. No new font imports anywhere.
> - **Icons** — Lucide React (`lucide-react` package), via the existing icon barrel `cms/packages/ui/src/components/icons.tsx`. No new icon libraries; if a glyph is missing, request it from Lucide first, fall back to a custom `<svg>` only if Lucide doesn't have it.
> - **Corner radius** — `0` (sharp). All `--radius-*` tokens are `0` per `tokens.css`. Never override with `rounded-md` etc.
> - **Light/dark theme** — `next-themes` + `.dark` class overrides; every new component must work in both themes.
> - **Design view ↔ Code view side-by-side** — `viewMode === "split"` preserved; regression test in Phase 5.2.

### Visual identity ESLint enforcement (§1.7)

Add a single ESLint rule pack `eslint-plugin-email-hub-visual-identity` (or extend existing) with the rules below. **Required before Phase 1 starts; runs in every PR's CI gate.**

| Rule | What it blocks | Why |
|---|---|---|
| `no-hardcoded-hex-colors` | Any `#[0-9a-fA-F]{3,8}` in JSX text, `style={...}` literals, Tailwind arbitrary `text-[#fff]`, or CSS-in-JS strings under `cms/apps/web/src/{app,components}/` | Forces semantic-token use. Allowlist for `lib/__fixtures__/` |
| `no-primitive-tailwind-colors` | `text-blue-500`, `bg-gray-100`, `border-red-300` etc. — primitive scale colors in JSX classNames | Forces semantic aliases (`text-foreground`, `bg-surface`) |
| `no-foreign-fonts` | `font-family` declarations with anything other than `var(--font-sans)`, `var(--font-mono)`, `var(--font-label)` | No new fonts |
| `no-foreign-icon-libs` | `import` from `react-icons`, `@heroicons/*`, `feather-icons`, etc. | Lucide-only |
| `no-radius-override` | `rounded-md`, `rounded-lg`, `rounded-full` in className; `border-radius: <non-zero>` in CSS | Sharp corners locked |
| `no-protocol-names-in-ui` (already in §1.6) | `DESIGN.md`, `A2UI`, `AG-UI`, `HITL`, `contract`, `evaluator`, `calibration` in JSX text | Email-Hub branding only |

Whitelisted directories: `cms/apps/web/src/lib/` (utility code may use raw hex internally, e.g. token mapping), `cms/apps/web/src/lib/**/__fixtures__/`, `cms/apps/web/docs/`, `*.test.tsx` snapshots.

## 2 · Backend impact

**Zero required.** All data the new modules consume already exists:

| Need | Existing source | Adapter strategy |
|---|---|---|
| Live agent progress | `useProgress` (`hooks/use-progress.ts`) → `/api/v1/progress/{id}` | `useAgUiStream` wraps it, translates to AG-UI envelope |
| Blueprint events | `useBlueprintRun` (`hooks/use-blueprint-run.ts`) | Stream chunks → AG-UI events |
| Agent decisions / structured output | Already returned by chat + blueprint endpoints in `*Decisions` shape | Client-side adapter `decisionsToA2UI()` synthesizes A2UI specs |
| 9-agent metadata | Hardcoded constants today + `use-agent-skills.ts` for skill metadata | New `lib/agents/registry.ts` (frontend constants) — **optional** future BE endpoint |
| QA results | `use-qa.ts` → `QAResultResponse` | Client-side `qaResultToA2UI()` adapter |
| Brand tokens for DESIGN.md | `tokens.css` (build-time) + `useBrandConfig` (runtime project overrides) | `tokensToDesignMd()` reads CSS variables at runtime |

**One recommended-but-optional backend tweak (not in this plan):** add `GET /api/v1/agents/registry` returning the 9 agents' versions and skill manifests. Saves ~80 lines of frontend constants. Defer to phase 11.

**Streaming caveat:** AG-UI events are polled at 1s via existing `useSmartPolling` rather than server-pushed SSE. Visually identical. Future phase can swap transport without touching consumers — `useAgUiStream` is the seam.

**Canonical A2UI v0.8 wire format** (binding for Phase 2):

- **Streaming via JSONL** with 4 message types: `surfaceUpdate`, `dataModelUpdate`, `beginRendering`, `deleteSurface`
- **Adjacency list components** (flat list, id-referenced — *not* nested tree)
- **Component wrapper:** `{id: "abc", component: {ComponentType: {…props}}}`
- **Standard catalog** (v0.8): `Text`, `Row`, `Column`, `Button`, `Card`, `Image`, `List`
- **Custom catalogs** allowed; server advertises via `catalogId` in `beginRendering`
- **Data binding:** `{literalString: "foo"}` or `{path: "/data/path"}`; data model lives in `dataModelUpdate`, decoupled from structure
- **No client-side code execution** — declarative only

### 12.1 — Symbiotic live system

| # | Rule | Where it lives | Verification |
|---|---|---|---|
| S1 | **Shared selection bus.** Selecting any email section on the canvas immediately propagates: crew rail highlights owning agents (gold ring), chat filters to that section's lineage, QA gate filters to scoped checks, DESIGN.md inspector shows the section's token usage. | `lib/selection/selection-context.tsx` — small React context exposing `{ sectionId, agentIds, source }`; consumed by 5 surfaces | Playwright: click section → assert 5 surfaces update within 1 frame |
| S2 | **Agents as collaborators, not services.** Active agents appear in the top-bar collaborator avatar stack alongside humans, distinguished by a small "AI" glyph and the agent's color. | S3 | **Action atomicity.** Applying any A2UI proposal triggers in one frame: optimistic canvas update → QA re-check enqueue → AG-UI event log → undo entry → toast. | S5 | **Live ripple cues.** When AG-UI events fire (`agent.complete`, `contract.failed`, `hitl.requested`), the affected zones get a 200ms tint pulse so causality is visible.
