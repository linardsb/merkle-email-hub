# Plan: 38.1 Figma Parser Data Fidelity Fixes

## Context

`_parse_node` in `figma/service.py` has 5 bugs producing inaccurate `DesignNode` data. All downstream stages (tree normalizer, converter, layout analyzer) depend on correct parsing. Transparent nodes render opaque, COMPONENT auto-layout is lost, stacked fills extract incompletely, non-linear gradients are dropped, and `visible: null` has unclear semantics.

## Research Summary

**Target file:** `app/design_sync/figma/service.py` (~1200 lines)
**Model:** `DesignNode` in `app/design_sync/protocol.py:94-128` — frozen dataclass with `opacity: float = 1.0`, `visible: bool = True`, `layout_mode: str | None`, `padding_*`, `item_spacing`, `counter_axis_spacing`, `fill_color`, `image_ref`

**Key helpers:**
- `_float_or_none(val)` at `service.py:146-148` — returns `float(val)` if numeric, else `None`
- `_gradient_midpoint_hex(stops)` at `service.py:130-143` — averages first/last stop
- `_compute_gradient_angle(handles)` at `service.py:151-162` — CSS angle from handle positions (linear-specific)
- `_parse_gradient_stops(stops, bg_hex)` at `service.py:164+` — converts to `(hex, position)` tuples

**Downstream consumers:**
- `tree_normalizer.py:109` — `if not child.visible or child.opacity <= 0.0` removes invisible nodes
- `converter_service.py:1019-1025` — maps `layout_mode` → `layout_direction` ("row"/"column")
- `converter.py` — `fill_color` → `bgcolor` attribute; `layout_mode` → cell generation
- `layout_analyzer.py:786+` — `image_ref` detection for background images; `fill_color` for section BG

**Node type mapping:** `COMPONENT`/`COMPONENT_SET` → `DesignNodeType.COMPONENT`, `INSTANCE` → `DesignNodeType.INSTANCE`. Auto-layout check at line 1075 uses `raw_type == "FRAME"` (the Figma API string), not the mapped enum.

**Gradient model:** `ExtractedGradient` in `protocol.py:54-62` already has `type: str` supporting `"linear" | "radial"`, but only `"linear"` is ever produced.

## Implementation Steps

### Step 1: Bug 1 — Opacity falsy trap (CRITICAL)

**File:** `service.py:1190`

**Current:**
```python
opacity=_float_or_none(node_data.get("opacity")) or 1.0,
```

**Fix:**
```python
opacity=_float_or_none(node_data.get("opacity")) if node_data.get("opacity") is not None else 1.0,
```

**Why:** `0.0 or 1.0` evaluates to `1.0` in Python. The `is not None` check avoids the falsy trap while defaulting missing values to `1.0`.

### Step 2: Bug 5 — `visible: null` handling (LOW)

**File:** `service.py:1189`

**Current:**
```python
visible=node_data.get("visible", True) is not False,
```

**Fix:**
```python
visible=node_data.get("visible") is not False,
```

**Why:** Drop the default — `None is not False` → `True`, which matches Figma's "inherit=visible" convention. Semantically clearer: we only hide explicitly hidden nodes.

### Step 3: Bug 2 — Auto-layout for COMPONENT/INSTANCE (HIGH)

**File:** `service.py:1075`

**Current:**
```python
if raw_type == "FRAME":
```

**Fix:**
```python
if raw_type in ("FRAME", "COMPONENT", "COMPONENT_SET", "INSTANCE"):
```

**Why:** Figma's COMPONENT, COMPONENT_SET, and INSTANCE nodes support auto-layout identically to FRAME. The rest of the extraction block (paddingTop/Right/Bottom/Left, itemSpacing, counterAxisSpacing, layoutMode) applies unchanged.

### Step 4: Bug 3 — Fill extraction order-dependence (MEDIUM)

**File:** `service.py:1114-1148`

Refactor the fill loop to collect ALL fills in a single pass without breaking early. Pseudocode:

```python
raw_fills = node_data.get("fills", [])
if isinstance(raw_fills, list):
    for fill_item in reversed(cast(list[Any], raw_fills)):
        if not isinstance(fill_item, dict):
            continue
        fi_d = cast(dict[str, Any], fill_item)
        if fi_d.get("visible") is False:
            continue
        fill_type = fi_d.get("type")

        if fill_type == "IMAGE":
            if node_type == DesignNodeType.VECTOR:
                node_type = DesignNodeType.IMAGE
            elif node_type == DesignNodeType.FRAME:
                raw_ref = fi_d.get("imageRef")
                if isinstance(raw_ref, str) and raw_ref:
                    image_ref = raw_ref
            continue  # always continue — don't break on IMAGE

        if fill_type != "SOLID":
            continue
        # ... extract solid fill (existing code) ...
        break  # break on first SOLID found (topmost wins)
```

**Key change:** Remove `break` after IMAGE reclassification for VECTOR. The loop continues to find SOLID fills underneath. The `break` on SOLID is fine — topmost solid fill should win.

### Step 5: Bug 4 — Radial/angular gradient support (LOW)

**File:** `service.py:820`

**Current:**
```python
if fill_d.get("type") == "GRADIENT_LINEAR":
```

**Fix:** Extend to handle `GRADIENT_RADIAL`, `GRADIENT_ANGULAR`, `GRADIENT_DIAMOND`:

```python
gradient_type = fill_d.get("type", "")
if gradient_type in ("GRADIENT_LINEAR", "GRADIENT_RADIAL", "GRADIENT_ANGULAR", "GRADIENT_DIAMOND"):
    stops = fill_d.get("gradientStops", [])
    if isinstance(stops, list) and len(stops) >= 2:
        midpoint = _gradient_midpoint_hex(cast(list[dict[str, Any]], stops))
        if midpoint and midpoint not in seen_hex:
            seen_hex.add(midpoint)
            gname = f"{node_name} (gradient midpoint)" if node_name else midpoint
            colors.append(ExtractedColor(name=gname, hex=midpoint, opacity=1.0))

        if gradients is not None:
            if gradient_type == "GRADIENT_LINEAR":
                handles = fill_d.get("gradientHandlePositions", [])
                angle = _compute_gradient_angle(handles)
                g_type = "linear"
            else:
                angle = 0.0  # angle not meaningful for radial/angular
                g_type = "radial"  # map all non-linear to "radial" for CSS
            parsed_stops = _parse_gradient_stops(stops, bg_hex=bg_hex)
            fallback = midpoint or "#808080"
            gradients.append(
                ExtractedGradient(
                    name=node_name or f"gradient-{len(gradients)}",
                    type=g_type,
                    angle=angle,
                    stops=tuple(parsed_stops),
                    fallback_hex=fallback,
                )
            )
```

**Notes:**
- `ExtractedGradient.type` already supports `"radial"` — no model changes needed
- Angular/diamond are rare; mapping to `"radial"` is acceptable for CSS output (midpoint color is the real value for email)
- `_compute_gradient_angle()` is linear-specific; skip it for non-linear types

- Existing pyright errors (26) are from untyped dict access — our changes should not increase this count
