# Plan: 50.4 Nested-Card Background (`container_bg` + `inner_bg`) — Gap 10

## Context

> **Concrete diff target:** see `.agents/plans/50-current-output-reference.md` § Section 1 ("Membership Card 1") for current vs expected markup excerpts. Acceptance criteria below check the markers identified there.

The white-card-on-coloured-wrapper pattern is the most-recurring structural misread in the LEGO conversion (sections #7, #8, #10–13, #16). The wrapper provides the colour (lime green); each inner card has its **own** white rounded background. Today's converter either:
- (a) treats the wrapper as one section (collapsed by Gap 1) → loses cards, or
- (b) treats inner sections independently → gives them all the wrapper's lime bg, losing the white surface.

Per `docs/architecture/opus-figma-to-html-process.md` Gap 10:
> *"The wrapper provides the colour, the inner card provides the surface for content... Component HTMLs gain an inner-table layer: `<td class="_outer"><table class="_inner">…</table></td>`."*

Phase 50.3 ships `container_bg` + `parent_wrapper_id`. This phase adds `inner_bg` + `inner_radius` and rewires renderer + 6 component HTMLs to support the two-layer `_outer`/`_inner` table structure.

**Detection of `inner_bg` from Figma data:**
- **Direct**: child FRAME's own `fills[0]` is non-default (e.g. white when wrapper is lime).
- **Indirect (PNG-sampled)**: child FRAME has empty fills, but full-design PNG (50.1) shows white pixels at the section's interior centroid while wrapper bg is lime. Sampling reuses `image_sampler.py` (50.2 infrastructure).

### Modify: section construction in `analyze_layout`

Detect `inner_bg`:

```python
def _detect_inner_bg(
    node: DesignNode,
    container_bg: str | None,
    global_design_image: bytes | None,
) -> tuple[str | None, float | None]:
    """Returns (inner_bg, inner_radius) if section has its own card surface."""
    # Direct: child FRAME has non-default fill different from container
    if node.fills:
        own_bg = _resolve_fill_hex(node.fills)
        if own_bg and own_bg != container_bg:
            radius = _resolve_corner_radius(node)
            return own_bg, radius

    # Indirect: PNG-sample interior centroid (when fill is empty but visual differs)
    if global_design_image and container_bg:
        sampled = _sample_section_centroid(node, global_design_image)
        if sampled and _hex_distance(sampled, container_bg) > 30:  # perceptual threshold
            radius = _resolve_corner_radius(node)
            return sampled, radius

    return None, None
```

Helper `_sample_section_centroid` reuses `image_sampler.sample_color_at` to read 5×5 pixel block at section's geometric center.

### Modify: `app/design_sync/component_matcher.py:1401-1469` — extend `_build_token_overrides`

```python
def _build_token_overrides(section: EmailSection) -> list[TokenOverride]:
    overrides: list[TokenOverride] = []

    # Outer wrapper background (Phase 50.3 + 50.4)
    if section.container_bg:
        overrides.append(TokenOverride("background-color", "_outer", section.container_bg))

    # Inner card background (Phase 50.4)
    if section.inner_bg:
        overrides.append(TokenOverride("background-color", "_inner", section.inner_bg))
    elif section.bg_color:
        # Fall back to old behavior when no inner card detected — preserves Phase 49 contract
        overrides.append(TokenOverride("background-color", "_outer", section.bg_color))

    # Inner card border radius (Phase 50.4)
    if section.inner_radius is not None:
        overrides.append(
            TokenOverride("border-radius", "_inner", f"{section.inner_radius:.0f}px")
        )

    # ... existing font, color, padding, CTA overrides (unchanged)
```

### Modify: `app/design_sync/component_renderer.py`

When `_inner` overrides are present, the rendered HTML must have a matching `_inner` table. Two implementation paths:

**Path A** (chosen): Add inner-table layer to 6 component HTMLs (article-card, zigzag-image-left/right, editorial-2, event-card, pricing-table).

**Path B** (rejected): Wrap dynamically in renderer. Loses semantic clarity in the templates; harder to maintain.

Per Path A, each component file gains an `_inner` table:

```html
<!-- article-card.html (excerpt) -->
<td class="_outer" style="padding: 16px 16px;">
  <table class="_inner" role="presentation" width="100%" cellpadding="0" cellspacing="0" border="0"
         style="border-collapse: separate; overflow: hidden;">
    <!-- existing rows go here -->
  </table>
</td>
```

The renderer's existing `_replace_css_prop_all` at component_renderer.py handles `border-radius` via class-fallback (Phase 49.4 expansion already supports this). When `inner_radius` override is emitted, the `_replace_*` handler picks up the `class="_inner"` and applies inline `border-radius:Npx; border-collapse:separate; overflow:hidden;`.

## Implementation Steps

### Step 1: Add `inner_bg` + `inner_radius` to `EmailSection`

Per spec.

### Step 2: Implement `_detect_inner_bg` in layout_analyzer

Two paths: explicit fill (cheap, deterministic) → PNG sampling fallback (when full-design PNG present). Both paths feed the same return shape.

### Step 3: Extend `_build_token_overrides`

Add the new emission. Crucial: `inner_bg` takes priority over `bg_color`; `bg_color` falls back to `_outer` only when no nested card detected.

### Step 4: Edit 6 component HTML files

Add `_inner` table layer. Verify with manual render of each component template — output should be identical to today when no inner_bg override is applied.

### Step 5: Wire `inner_bg` into renderer's existing class-replace handler

Phase 49.4 already supports class-fallback for `_inner` target — verify by grepping `component_renderer.py` for `_inner` references. If not, add to the dispatch table.

## Out of Scope

- Inner-card width matching (Rule 11 in 50.5 — needs this plan's nested structure first)
- Cards with multiple nesting levels (>2 layers — address only if real designs need it)
