# Plan: 34.2 Structured CRAG Observability Logging

## Context

CRAG logs `agents.crag.correction_applied` on success and `agents.crag.output_too_short` on length failure — but there's no visibility into *what* was fixed vs. *what* regressed. Operators need structured fields for dashboards/alerting: which CSS properties fail, which fallback techniques regress, whether rejection rate is climbing. This is logging-only — no new endpoints, no PII.

**Note:** The task spec says `correction_rejected` was "added in 34.1" but it does not exist yet. This plan adds both the rejection log and the enhanced success/detection logs.

### Step 4: Rename + enhance `correction_applied` → `correction_accepted` (validation_loop.py:134-139)

```python

corrections = [str(issue["property_id"]) for issue in qualifying]
```

**Note:** `post_issues` and `issues_fixed` require a re-scan (`unsupported_css_in_html(corrected)`) which adds latency. The spec includes them but the current code does not re-scan. Adding re-scan:

```python
# After line 127 (sanitize), before the length check
post_issues_raw = unsupported_css_in_html(corrected) if corrected and len(corrected) >= 50 else []
post_qualifying = [
    i for i in post_issues_raw if severity_order.get(str(i["severity"]), 2) <= threshold
]
```

Then the accepted log becomes:

```python
logger.info(
    "agents.crag.correction_accepted",
    pre_issues=len(qualifying),
    post_issues=len(post_qualifying),
    issues_fixed=len(qualifying) - len(post_qualifying),
    corrections=corrections,
    original_length=len(html),
    corrected_length=len(corrected),
)
```

## Preflight Warnings

- `_make_settings()` in tests returns MagicMock — caplog tests need real structlog setup or `caplog` fixture
- Tests patch `unsupported_css_in_html` at module level — re-scan in Step 4 needs the mock to be called twice (once for detection, once for post-correction)
- `structlog` caplog integration: use `structlog.testing.capture_logs()` context manager or pytest `caplog` with structlog bound logger
