# Plan: Blueprint Run Trigger & Pipeline Visualisation (Phase 10.3)

## Context
The blueprint engine (Phase 4.13, enhanced through Phase 9) orchestrates multi-agent pipelines with self-correction, QA gating, audience context, and failure pattern propagation — but it's currently CLI-only. The workspace chat sidebar calls individual agent endpoints, bypassing all intelligence layers. This feature exposes the full blueprint engine via a workspace dialog with real-time pipeline progress visualisation.

### Step 3: Hook (`cms/apps/web/src/hooks/use-blueprint-run.ts`)

```typescript
"use client";

import { useCallback, useState } from "react";
import { authFetch } from "@/lib/auth-fetch";
import type { BlueprintRunResponse } from "@email-hub/sdk";

interface UseBlueprintRunOptions {
  projectId: number;
}

interface BlueprintRunParams {
  blueprint_name: string;
  brief: string;
  initial_html?: string;
  persona_ids?: number[];
}

export function useBlueprintRun({ projectId }: UseBlueprintRunOptions) {
  const [isRunning, setIsRunning] = useState(false);
  const [result, setResult] = useState<BlueprintRunResponse | null>(null);
  const [error, setError] = useState<string | null>(null);

  const run = useCallback(
    async (params: BlueprintRunParams) => {
      setIsRunning(true);
      setError(null);
      setResult(null);

      try {
        const res = await authFetch("/api/v1/blueprints/run", {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify({
            ...params,
            options: { project_id: projectId },
          }),
        });

        if (!res.ok) {
          const body = await res.json().catch(() => null);
          throw new Error(body?.error ?? "Blueprint run failed");
        }

        const data: BlueprintRunResponse = await res.json();
        setResult(data);
        return data;
      } catch (err) {
        const message = err instanceof Error ? err.message : "Blueprint run failed";
        setError(message);
        return null;
      } finally {
        setIsRunning(false);
      }
    },
    [projectId],
  );

  const reset = useCallback(() => {
    setResult(null);
    setError(null);
    setIsRunning(false);
  }, []);

  return { run, isRunning, result, error, reset };
}
```

**Design note:** The backend `POST /api/v1/blueprints/run` is synchronous — it returns the full `BlueprintRunResponse` when complete. No polling/WebSocket needed for v1. The dialog shows a loading state during the request, then renders the full pipeline result. Future enhancement could add SSE streaming for real-time node-by-node updates.

### Step 6: Dialog Component (`cms/apps/web/src/components/workspace/blueprint-run-dialog.tsx`)

Create the dialog with these sections:

**Props:**

**UI Structure:**
1. **Input Phase** (before run starts):
   - Blueprint selector — only "campaign" for now, shown as a radio/select with name + description
   - Textarea for campaign brief (required, min length validation)
   - Checkbox: "Include current editor HTML as starting point" (uses `currentHtml` prop)
   - "Run Pipeline" primary button

2. **Running Phase** (`isRunning === true`):
   - Loading spinner with "Running pipeline..." text
   - Disabled inputs

3. **Result Phase** (after `result` is set):
   - **Status Banner**: coloured banner showing run status (`completed` = green, `completed_with_warnings` = yellow, `needs_review` = amber, `cost_cap_exceeded` = red)
   - **Pipeline Timeline**: Vertical list of `progress` entries:
     - Each node shows: icon (Bot for agentic, Cog for deterministic), node name (formatted), status badge, duration, iteration count if > 0
     - Colour-coded: success = `text-chart-2`, failed = `text-destructive`, skipped = `text-muted-foreground`
     - QA gate nodes get special treatment — show "8/10 passed" summary
     - Self-correction loops are visually grouped (same qa_gate appears multiple times with incrementing iteration)
   - **Handoff History** (collapsible):
     - Each handoff shows: agent name badge, decisions list, warnings (if any, in destructive colour), component refs, confidence bar
   - **Routing Decisions** (if any skipped nodes):
     - Show skipped nodes with reason
   - **Audience Summary** (if present):
     - Muted text showing audience context used
   - **Token Usage**: Small footer showing total tokens consumed
   - **Actions**: "Apply to Editor" primary button + "Close" outline button

**Semantic Tailwind tokens to use:**
- Status badges: `bg-accent text-accent-foreground` (success), `bg-destructive/10 text-destructive` (failed), `bg-muted text-muted-foreground` (skipped/pending)
- Timeline connector: `border-border`
- Cards: `bg-card text-card-foreground border-border`
- Headings: `text-foreground`
- Body text: `text-muted-foreground`
- Confidence bar fill: `bg-primary`

**UI Components to import:**
- `Dialog`, `DialogContent`, `DialogHeader`, `DialogTitle`, `DialogDescription`, `DialogFooter` from ui
- `Button` from ui
- `Badge` from ui
- `Textarea` from ui (or use standard HTML textarea with Tailwind)
- `Skeleton` from ui (for loading state)
- `Label` from ui
- Icons from `lucide-react`: `Bot`, `Cog`, `CheckCircle2`, `XCircle`, `SkipForward`, `ChevronDown`, `ChevronUp`, `Zap`, `Clock`, `AlertTriangle`

**Key implementation details:**
- Dialog width: `sm:max-w-[40rem]` (between detail 28rem and the 48rem compatibility brief)
- Max height: `max-h-[80vh] overflow-y-auto` (matches compatibility brief pattern)
- All text via `useTranslations("blueprintRun")`
- Format node names: replace underscores with spaces, title case (e.g., `"recovery_router"` → `"Recovery Router"`)
- Format duration: show as seconds with 1 decimal for > 1000ms, otherwise ms
- Confidence display: percentage with colour (>0.8 = accent, 0.5-0.8 = warning-like muted, <0.5 = destructive)
