# Plan: 10.1 Project Target Clients Selector UI

## Context
`target_clients` drives audience profiles (9.1), onboarding subgraphs (9.5), failure pattern filtering (9.4), and blueprint route selection (9.6). The backend fully supports `target_clients` on `ProjectCreate`, `ProjectUpdate`, and `ProjectResponse` schemas, and auto-triggers onboarding subgraph generation when set. But the frontend has **no UI** for selecting target clients — the SDK types are stale (missing `target_clients`), and the create-project dialog doesn't include a client selector.

### Key Findings
- **Backend**: `target_clients: list[ClientId] | None` on `ProjectCreate`, `ProjectUpdate`, `ProjectResponse` — all wired
- **SDK types**: `types.gen.ts` is stale — `ProjectCreate`, `ProjectUpdate`, `ProjectResponse` all missing `target_clients`
- **Ontology**: 25 email clients defined in `app/knowledge/ontology/data/clients.yaml` — `EmailClient` dataclass has `id`, `name`, `family`, `platform`, `engine`, `market_share`
- **No ontology clients endpoint exists** — need to add `GET /api/v1/ontology/clients` OR use a static list. Decision: **add a backend endpoint** since the ontology data is server-authoritative and may change with Can I Email sync.
- **Create dialog**: `create-project-dialog.tsx` has the form but no `target_clients` field
- **Project cards**: `projects/page.tsx` displays name, description, status, date — no client badges

**`app/knowledge/ontology/routes.py`** — Add endpoint after existing imports and before `get_competitive_report`:

```python
from app.knowledge.ontology.schemas import EmailClientResponse

@router.get("/clients", response_model=list[EmailClientResponse])
@limiter.limit("30/minute")
async def list_email_clients(
    request: Request,  # noqa: ARG001
    current_user: Annotated[User, Depends(get_current_user)],  # noqa: ARG001
) -> list[EmailClientResponse]:
    """List all email clients from the ontology registry."""
    from app.knowledge.ontology.registry import load_ontology

    registry = load_ontology()
    return [
        EmailClientResponse(
            id=c.id,
            name=c.name,
            family=c.family,
            platform=c.platform,
            engine=c.engine.value,
            market_share=c.market_share,
        )
        for c in registry.clients
    ]
```

### Step 3: Frontend — SWR hooks

**`cms/apps/web/src/hooks/use-email-clients.ts`**:

```typescript
"use client";

import useSWR from "swr";
import { fetcher } from "@/lib/swr-fetcher";
import type { EmailClientResponse } from "@email-hub/sdk";

export function useEmailClients() {
  return useSWR<EmailClientResponse[]>(
    "/api/v1/ontology/clients",
    fetcher,
    { revalidateOnFocus: false, dedupingInterval: 300_000 } // 5min cache — static-ish data
  );
}
```

**`cms/apps/web/src/hooks/use-update-project.ts`**:

```typescript
"use client";

import useSWRMutation from "swr/mutation";
import { mutationFetcher } from "@/lib/mutation-fetcher";
import type { ProjectResponse, ProjectUpdate } from "@email-hub/sdk";

export function useUpdateProject(projectId: number) {
  return useSWRMutation<ProjectResponse, Error, string, ProjectUpdate>(
    `/api/v1/projects/${projectId}`,
    async (url: string, { arg }: { arg: ProjectUpdate }) => {
      const { authFetch } = await import("@/lib/auth-fetch");
      const res = await authFetch(url, {
        method: "PATCH",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(arg),
      });
      if (!res.ok) throw new Error("Failed to update project");
      return res.json() as Promise<ProjectResponse>;
    }
  );
}
```
