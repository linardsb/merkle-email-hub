# Plan: 44.4 Adversarial Agent Evaluation Pass

## Context

The eval pipeline tests agents against well-formed inputs. Production inputs are adversarial by nature: Word-pasted text, 200+ layer Figma designs, 5-level ESP conditionals. This phase adds an adversarial evaluation stage generating hostile inputs (long strings, RTL text, nested conditionals, missing assets, extreme widths, emoji-heavy, malformed HTML) to surface agent weaknesses before production.

## Implementation Steps

### Step 1: Schema — `AdversarialCase` dataclass in `schemas.py`

Add after `SkillFilePatch` (line 192):

```python
ADVERSARIAL_ATTACK_TYPES = (
    "long_string",
    "rtl_injection",
    "nested_conditionals",
    "missing_assets",
    "extreme_width",
    "emoji_heavy",
    "malformed_html",
)

@dataclass(frozen=True)
class AdversarialCase:
    """A hostile test input designed to break agent output."""
    name: str                    # e.g., "scaffolder_long_heading"
    agent: str                   # one of 9 agent names
    attack_type: str             # one of ADVERSARIAL_ATTACK_TYPES
    input_html: str              # hostile input content
    description: str             # what this tests
    expected_behavior: str = ""  # expected agent response
```

### Step 2: Core module — `adversarial.py`

```python
"""Adversarial test case generator and YAML loader.

Generates hostile inputs across 7 attack types for all 9 agents.
Cases are either generated programmatically or loaded from YAML fixtures.
"""
```

Key functions:

| Function | Returns | Logic |
|----------|---------|-------|
| `generate_adversarial_cases(agent)` | `list[AdversarialCase]` | Dispatches to 7 `_*_cases()` generators, filters by agent applicability |
| `load_yaml_cases(agent)` | `list[AdversarialCase]` | Reads `test_cases/adversarial/{agent}.yaml`, validates schema |
| `get_all_cases(agent)` | `list[AdversarialCase]` | Merges generated + YAML, deduplicates by name |
| `_long_string_cases(agent)` | `list[AdversarialCase]` | 500-char heading, 10KB paragraph, long alt text |
| `_rtl_injection_cases(agent)` | `list[AdversarialCase]` | Arabic/Hebrew mixed with LTR content |
| `_nested_conditional_cases(agent)` | `list[AdversarialCase]` | 5-level Liquid `{% if %}` nesting, AMPscript `%%[IF]%%` nesting |
| `_missing_asset_cases(agent)` | `list[AdversarialCase]` | Broken `<img src="">`, missing font-family refs |
| `_extreme_dimension_cases(agent)` | `list[AdversarialCase]` | 200px mobile, 1200px ultra-wide viewport hints in brief/HTML |
| `_emoji_cases(agent)` | `list[AdversarialCase]` | Emoji in headings, alt text, CTA buttons, subject lines |
| `_malformed_html_cases(agent)` | `list[AdversarialCase]` | Unclosed `<table>`, nested `<a>` inside `<a>`, `<td>` without `<tr>` |

**Agent applicability matrix** — not all attack types apply to all agents:

| Attack type | HTML agents | Content | Knowledge | Innovation |
|-------------|-------------|---------|-----------|------------|
| long_string | ✓ | ✓ (long text input) | ✓ (long question) | ✓ (long technique desc) |
| rtl_injection | ✓ | ✓ | ✓ | ✗ |
| nested_conditionals | ✓ (relevant to personalisation, outlook_fixer) | ✗ | ✗ | ✗ |
| missing_assets | ✓ | ✗ | ✗ | ✗ |
| extreme_width | ✓ | ✗ | ✗ | ✓ |
| emoji_heavy | ✓ | ✓ | ✓ | ✗ |
| malformed_html | ✓ | ✗ | ✗ | ✗ |

HTML agents = scaffolder, dark_mode, outlook_fixer, accessibility, personalisation, code_reviewer.

Each generator produces **2 cases per applicable agent** → minimum 10+ cases for HTML agents, 4-6 for non-HTML agents.

**HTML generation rules:** All adversarial HTML uses `<table>/<tr>/<td>` layout (no div/p for structure). Malformed cases break rules intentionally but within a table skeleton. All HTML passes through `sanitize_html_xss()` before reaching agents — same as production.

### Step 3: YAML Fixtures — `test_cases/adversarial/{agent}.yaml`

Each file contains 2-3 curated adversarial cases that are hard to generate programmatically:

```yaml
# test_cases/adversarial/scaffolder.yaml
- name: scaffolder_word_paste_formatting
  attack_type: malformed_html
  input_html: |
    <table><tr><td>
    <h1 style="mso-style-type:export-only;font-family:'Times New Roman'">
    <span style="font-size:14.0pt;line-height:115%">Pasted from Word</span>
    </h1><o:p></o:p>
    </td></tr></table>
  description: "Word-pasted HTML with MSO namespace pollution"
  expected_behavior: "Strip mso-style-type, normalize fonts, preserve heading"

- name: scaffolder_200_layer_brief
  attack_type: long_string
  input_html: ""
  description: "Brief requesting 50+ sections"
  expected_behavior: "Truncate or group sections, no timeout"
```

For knowledge agent, `input_html` contains the question text. For content agent, it contains the text to process. Field reuse keeps schema simple.

### Step 7: Error Analysis Integration — `error_analysis.py`

Extend `build_analysis_report()` to include adversarial metrics when adversarial verdict files exist:

```python
# After loading regular verdicts, also load adversarial
adv_verdicts = []
for f in sorted(verdicts_path.glob("*_adversarial_verdicts.jsonl")):
    adv_verdicts.extend(load_verdicts(f))

if adv_verdicts:
    adv_pass_rates = compute_pass_rates(adv_verdicts)
    adv_overall = sum(1 for v in adv_verdicts if v.get("overall_pass")) / len(adv_verdicts)
    report["adversarial"] = {
        "total": len(adv_verdicts),
        "overall_pass_rate": adv_overall,
        "pass_rates": adv_pass_rates,
        "status": "FAIL" if adv_overall < 0.40 else "WARN" if adv_overall < 0.60 else "PASS",
    }
```

In the `main()` CLI output, log adversarial status:

```python
if "adversarial" in report:
    adv = report["adversarial"]
    status_label = {"PASS": "✓", "WARN": "⚠", "FAIL": "✗"}[adv["status"]]
    logger.info(f"Adversarial: {adv['overall_pass_rate']:.1%} ({adv['total']} cases) {status_label}")
```

## Preflight Warnings

- `runner.py:560-599` uses a long if/elif chain for agent dispatch — extending it is straightforward but fragile. Consider extracting to a dict after this phase.
