# Plan: Phase 32.11 — Per-Client Skill Overlays

## Context

Different brands need different agent *behaviors* — not just different data. The centralized client matrix (32.1) provides email client rendering facts, but brand-specific behavioral guidance (layout patterns, tone, color mappings) currently only flows through L4 knowledge (unstructured RAG). Per-client L3 overlays provide structured, version-controlled, budget-aware behavioral customization within the existing skill loading pipeline. Supports agency use cases (single deployment, multiple clients).

### Current Skill Loading Flow

```
BaseAgentService.process(request)
  → _detect_skills_from_request(request)         # returns list[str] of skill keys
  → build_system_prompt(relevant_skills, output_mode)  # each agent's prompt.py
      → _base_system_prompt(output_mode)          # L1+L2 from SKILL.md
      → for skill_key in relevant_skills:
          → SKILL_FILES[skill_key] → filename
          → _load_skill_file(filename) → raw_content
          → parse_skill_meta(raw_content) → (SkillMeta, body)
          → should_load_skill(meta, cumulative_cost, remaining_budget, budget_max)
          → parts.append(f"--- REFERENCE: {skill_key} ---\n\n{body}")
```

### Blueprint Node Pattern (DarkModeNode example, `dark_mode_node.py:53-64`)

```python
async def execute(self, context: NodeContext) -> NodeResult:
    relevant_skills = detect_relevant_skills(context.html)       # from prompt.py
    system_prompt = build_system_prompt(relevant_skills)          # from prompt.py
    # ... build messages, call LLM
```

Nodes import agent-specific `detect_relevant_skills` and `build_system_prompt` directly — NOT via `BaseAgentService`. This means overlay integration happens in **two places**: `base.py` (for service calls) and each node's `prompt.py` (for blueprint calls).

### Existing Frontmatter Format (from `skill_loader.py:19-29`)

```yaml
---
token_cost: 450
priority: 2
---
```

Parsed by `parse_skill_meta()` with regex `^---\s*\n(.*?)\n---\s*\n`. Simple line-by-line key:value extraction (no YAML library dependency). Defaults: `token_cost=500`, `priority=2`.

### Client Organization Model (`projects/models.py:12-24`)

- `ClientOrg`: `id`, `name`, `slug` (unique), `is_active`
- `Project.client_org_id` → FK to `ClientOrg.id`
- Use `client_org.slug` as `client_id` for overlay directory naming (e.g., `"acme-corp"`)

### Context Threading (`engine.py:645-757`)

`_build_node_context()` progressively hydrates `context.metadata` with ~17 layers. `project_id` is stored as `self._project_id` but NOT currently injected into node context. Adding `client_id` follows the same pattern as existing metadata injection.

Add cached discovery function:

```python
@lru_cache(maxsize=128)
def discover_overlays(agent_name: str, client_id: str) -> tuple[OverlayMeta, ...]:
    """Discover overlay skill files for a client+agent pair.

    Scans data/clients/{client_id}/agents/{agent_name}/skills/*.md.
    Returns tuple (hashable for cache). Empty tuple if no overlays found.
    """
    overlay_dir = _OVERLAYS_BASE / client_id / "agents" / agent_name / "skills"
    if not overlay_dir.is_dir():
        return ()
    overlays = []
    for md_file in sorted(overlay_dir.glob("*.md")):
        raw = md_file.read_text(encoding="utf-8")
        meta, body = parse_overlay_meta(raw)
        overlays.append(OverlayMeta(
            token_cost=meta.token_cost, priority=meta.priority,
            overlay_mode=meta.overlay_mode, replaces=meta.replaces,
            client_id=client_id, content=body,
            source_path=str(md_file.relative_to(_OVERLAYS_BASE)),
        ))
    return tuple(overlays)
```

Add overlay application function:

```python
def apply_overlays(
    skill_parts: list[str],
    loaded_skills: set[str],
    overlays: tuple[OverlayMeta, ...],
    cumulative_cost: int,
    remaining_budget: int,
    budget_max: int,
) -> tuple[list[str], int, list[str]]:
    """Apply client overlays to the skill loading pipeline.

    Returns: (updated_parts, updated_cumulative_cost, overlay_names_loaded)
    """
    overlay_names: list[str] = []

    for overlay in overlays:
        # Handle "replace" mode: remove the named core skill
        if overlay.overlay_mode == "replace" and overlay.replaces:
            # Find and remove the replaced skill's content from parts
            tag = f"--- REFERENCE: {overlay.replaces} ---"
            skill_parts = [p for p in skill_parts if tag not in p]
            loaded_skills.discard(overlay.replaces)

        # Budget check using same logic as core skills
        meta = SkillMeta(token_cost=overlay.token_cost, priority=overlay.priority)
        if not should_load_skill(meta, cumulative_cost, remaining_budget, budget_max):
            continue

        cumulative_cost += overlay.token_cost
        overlay_name = Path(overlay.source_path).stem
        label = f"overlay:{overlay.client_id}/{overlay_name}"
        skill_parts.append(f"\n\n--- REFERENCE: {label} ---\n\n{overlay.content}")
        overlay_names.append(label)

    return skill_parts, cumulative_cost, overlay_names
```

### Step 7: Create `scripts/validate-overlays.py`

Validates all overlay files in `data/clients/`:

1. **Frontmatter validation**: required fields present, `overlay_mode` ∈ `{"extend", "replace"}`, `token_cost` is positive int, `priority` ∈ `{1, 2, 3}`
2. **Replace reference check**: when `overlay_mode="replace"`, verify `replaces` value exists in target agent's `SKILL_FILES` dict (import each agent's `prompt.py`)
3. **Budget check**: overlay `token_cost` + sum of non-replaced core skills ≤ agent's `skill_docs_max`
4. **Conflict detection**: no two overlays for same client+agent both replacing the same core skill
5. **Exit code 0** on success, **1** on any failure (for CI/pre-commit integration)
