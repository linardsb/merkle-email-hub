# Plan: 44.5 Operational Runbooks

## Context

The codebase has excellent developer docs (ARCHITECTURE.md, 8 ADRs, CLAUDE.md, deploy-hetzner-coolify.md) but zero operational documentation. When production breaks at 2 AM, the responder needs step-by-step procedures with commands and decision trees — not architectural diagrams. Existing operational scripts (`backup-db.sh`, `restore-db.sh`, `safe_alembic.sh`, `startup.sh`) are undocumented.

## Implementation Steps

### Step 1: Create `docs/operations/` directory

```bash
mkdir -p docs/operations
```

### Step 2: `docs/operations/deployment-checklist.md`

Sections:
1. **Pre-Deploy Checks** — `make check-full`, `make flag-audit`, verify migration count (`uv run alembic history | wc -l`), review feature-flags.yaml for stale flags
2. **Database Migration** — `uv run alembic upgrade head`, verify with `uv run alembic current`, expected output format. Reference `scripts/safe_alembic.sh` for production use
3. **Service Startup Order** — db (healthy) → redis (healthy) → migrate (completed) → app (healthy) → cms (healthy) → nginx (healthy). Commands: `docker compose up -d`, monitor: `docker compose logs -f`
4. **Post-Deploy Verification** — curl each health endpoint, expected responses, smoke test: create project, run scaffolder, verify email output
5. **Rollback Procedure** — `scripts/safe_alembic.sh downgrade -1 --i-know-what-i-am-doing` (auto-creates backup), redeploy previous image tag, verify rollback with `uv run alembic current`
6. **Environment Promotion** — dev → staging → production checklist (secrets rotation, CORS origins, SSL certs)

### Step 3: `docs/operations/disaster-recovery.md`

Sections:
1. **PostgreSQL Backup** — `scripts/backup-db.sh` usage, cron schedule (`0 */6 * * *`), backup location (`/data/backups/email-hub/`), retention (10 backups, ~60hrs)
2. **PostgreSQL Restore** — `scripts/restore-db.sh [file]` or auto-select latest, what it does (truncate → restore → verify row counts), expected output
3. **WAL Archiving** (recommended setup) — `archive_mode=on`, `archive_command`, PITR restore steps
4. **Redis Recovery** — what's in Redis (transient: rate limits, cache, pub/sub; persistent: feature flags), RDB snapshot config, AOF recommended settings, cache warm-up (flags auto-reload, rate limits reset naturally, section cache rebuilds on demand)
5. **State Classification** — table: PostgreSQL (users, projects, components, templates, evals, migrations) vs Redis (sessions, rate limits, flags, cache, pub/sub)
6. **RTO Targets** — DB <1hr (restore from dump), Redis <5min (restart + flag reload), app <10min (docker compose restart)

### Step 4: `docs/operations/incident-response.md`

Sections:
1. **Severity Classification** — S1 (platform down, all health checks failing), S2 (feature broken, partial health failure), S3 (degraded performance, latency >5s p99), S4 (cosmetic, no user impact)
2. **Triage Flowchart** (text decision tree):
   - `curl /health/ready` → if 503: check `/health/db` and `/health/redis` individually
   - `docker compose ps` → identify unhealthy services
   - `docker compose logs <service> --tail 50` → identify error
   - `git log --oneline -5` → check recent deploys
3. **Common Incidents** — each with symptoms, diagnosis, resolution:
   - Migration failure: app won't start, migrate container exit code 1 → check `docker compose logs migrate`, fix migration, re-run
   - Redis OOM: rate limiting falls back to in-memory, feature flags use defaults → `docker compose exec redis redis-cli INFO memory`, check maxmemory
   - AI provider outage: 502 on generation endpoints → check fallback chain config (`AI__FALLBACK_CHAINS`), verify API key validity
   - Database connection exhaustion: 503 on `/health/db` → check `pg_stat_activity` count vs `POOL_SIZE + MAX_OVERFLOW`, restart app
   - Eval pipeline disagreement: >20% flip rate on re-judge → check `scripts/eval-compare-verdicts.py`, manual verdict override
4. **Post-Incident** — write brief in `docs/incidents/YYYY-MM-DD-title.md` (template provided), update affected runbook

### Step 5: `docs/operations/performance-tuning.md`

Sections:
1. **PostgreSQL** — `DATABASE__POOL_SIZE` (rule: 2× CPU cores), `DATABASE__POOL_MAX_OVERFLOW` (burst headroom), `DATABASE__POOL_RECYCLE` (1hr default, reduce if stale connections), monitoring: `SELECT count(*) FROM pg_stat_activity`, `pg_stat_statements` for slow queries
2. **Redis** — `maxmemory` (default 128M in docker, increase for heavy cache use), `maxmemory-policy allkeys-lru` (recommended), monitor: `redis-cli INFO memory`, `redis-cli INFO stats` (keyspace hits/misses)
3. **Gunicorn** — `WEB_CONCURRENCY` (rule: 2× CPU + 1), `--timeout` (120s default, reduce for fast-fail), `--max-requests` (1000, prevents memory leaks), `--graceful-timeout` (30s)
4. **Uvicorn** — `--workers` (same as WEB_CONCURRENCY), `--limit-concurrency` (default unlimited, set to prevent overload), `--backlog` (default 2048)
5. **Application** — `RATE_LIMIT__*` tuning table, `AI__TOKEN_BUDGET_*` settings, `AI__DAILY_QUOTA` per-user throttle, `AI__COST_GOVERNOR_ENABLED` monthly budget cap
