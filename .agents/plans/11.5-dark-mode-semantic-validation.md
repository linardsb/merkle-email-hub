### Step 1: Create `dark_mode_parser.py` — Standalone Parser Module

**Pattern:** Follow `mso_parser.py` — standalone module with public API, caching, used by both QA check and Dark Mode agent.

**File:** `app/qa_engine/dark_mode_parser.py`

**Public API:**
```python
@dataclass(frozen=True)
class DarkModeValidationResult:
    issues: list[str]
    meta_tags: MetaTagInfo           # parsed meta tag state
    media_queries: list[MediaQueryBlock]  # parsed @media blocks
    outlook_selectors: list[OutlookSelector]  # [data-ogsc]/[data-ogsb] blocks
    color_pairs: list[ColorPair]     # light→dark color mappings found
    has_image_swap: bool             # CSS show/hide or <picture> pattern
    has_1x1_trick: bool              # Outlook 1x1 pixel background trick

def validate_dark_mode(html: str) -> DarkModeValidationResult: ...
def get_cached_result(html: str) -> DarkModeValidationResult: ...
```

**Internal Validators (5):**

1. **`_parse_meta_tags(doc, raw_html)`** — Extract and validate:
   - `<meta name="color-scheme" content="...">` — must be in `<head>`, content must include "dark"
   - `<meta name="supported-color-schemes" content="...">` — legacy companion tag
   - CSS `color-scheme: light dark` property in `<style>` block (`:root` or `html`)
   - Return `MetaTagInfo(has_color_scheme, has_supported_color_schemes, has_css_color_scheme, color_scheme_in_head, content_value)`

2. **`_parse_media_queries(raw_html)`** — Extract `@media (prefers-color-scheme: dark)` blocks:
   - Use regex to find blocks and their CSS content (don't need full CSS parser — email CSS is simple)
   - Parse individual declarations: property name, value, `!important` presence
   - Track which color properties are present (`color`, `background-color`, `background`, `border-color`, `border`)
   - Flag empty blocks (no declarations), blocks with no color properties, blocks missing `!important`
   - Return list of `MediaQueryBlock(content, declarations, has_color_props, has_important, is_empty)`

3. **`_parse_outlook_selectors(raw_html)`** — Extract `[data-ogsc]` and `[data-ogsb]` selector blocks:
   - Find all CSS rules using these attribute selectors
   - Validate they contain actual color declarations (not empty)
   - Check `[data-ogsc]` rules have foreground properties (`color`)
   - Check `[data-ogsb]` rules have background properties (`background-color`, `background`)
   - Return list of `OutlookSelector(selector_type, rules, has_declarations)`

4. **`_extract_color_pairs(doc, raw_html, media_queries)`** — Light↔dark color mapping:
   - Extract inline style colors from DOM elements (the "light mode" values)
   - Extract corresponding dark mode values from `@media (prefers-color-scheme: dark)` block
   - Match by CSS class: element with `class="foo"` → `.foo { color: X }` in media query
   - Return list of `ColorPair(selector, property, light_value, dark_value, contrast_ratio)`
   - Contrast calculation: convert hex to luminance, compute WCAG 2.1 ratio

5. **`_detect_image_patterns(doc, raw_html)`** — Image handling detection:
   - `<picture>` + `<source media="(prefers-color-scheme: dark)">` (Apple Mail swap)
   - CSS show/hide pattern: `.light-img { display: none }` + `.dark-img { display: block }` in media query
   - 1x1 pixel background trick: `background-image: url(...)` with `background-color` on same element (in `<td>`, `<table>`, `<body>`)
   - Return booleans for each pattern detected

**Helper functions:**
- `_hex_to_luminance(hex_color: str) -> float` — Convert hex color to relative luminance
- `_contrast_ratio(l1: float, l2: float) -> float` — WCAG contrast ratio formula
- `_parse_css_color(value: str) -> str | None` — Extract hex color from CSS value (handle rgb(), named colors for common ones)

**Caching:** Same pattern as `mso_parser.py` — `functools.lru_cache` on hash of HTML.

def dm_color_scheme_meta(
    doc: HtmlElement, raw_html: str, config: QACheckConfig | None
) -> tuple[list[str], float]:
    deduction = float(_dm_param(config, "deduction_missing_color_scheme", 0.15))
    result = get_cached_dm_result(raw_html)

    return ([], 0.0)

### Step 4: Rewrite `dark_mode.py` — Rule Engine Integration

**Pattern:** Identical to `accessibility.py` (77 lines). The check becomes a thin wrapper.

```python
"""Dark mode semantic validation — DOM-parsed + CSS-parsed checks.

Task 11.5: Upgraded from presence-only to semantic validation.
"""

from pathlib import Path

from lxml import html as lxml_html

from app.qa_engine.check_config import QACheckConfig
from app.qa_engine.rule_engine import RuleEngine, load_rules
from app.qa_engine.schemas import QACheckResult

import app.qa_engine.custom_checks  # noqa: F401 — registers custom fns

_RULES_PATH = Path(__file__).parent.parent / "rules" / "dark_mode.yaml"
_rules = load_rules(_RULES_PATH)


class DarkModeCheck:
    """Semantic dark mode validation via YAML rule engine."""

    name = "dark_mode"

    async def run(self, html: str, config: QACheckConfig | None = None) -> QACheckResult:
        doc = lxml_html.document_fromstring(html)
        engine = RuleEngine(_rules)
        issues, total_deduction = engine.evaluate(doc, html, config)

        score = max(0.0, round(1.0 - total_deduction, 2))
        passed = score >= 0.5  # configurable threshold if needed

        return QACheckResult(
            check_name=self.name,
            passed=passed,
            score=score,
            details="; ".join(issues) if issues else None,
            severity="warning",
        )
```

## Architecture Decisions

1. **Separate parser module** (`dark_mode_parser.py`) — same pattern as `mso_parser.py`. Enables the Dark Mode agent to import `validate_dark_mode()` directly for post-generation validation (task 11.14).

2. **Regex-based CSS parsing** — email CSS is simple enough that regex works reliably. No dependency on `cssutils` or `tinycss2` (those are tasks 11.10). Use regex to extract `@media (prefers-color-scheme: dark) { ... }` blocks and parse declarations within.

3. **Color contrast via luminance** — implement the WCAG 2.1 relative luminance formula inline (it's ~20 lines). No external dependency needed. Handle hex colors (`#fff`, `#ffffff`), rgb() values, and the most common named colors (white, black, transparent).

4. **Condensed L3 reference** — the full 882-line doc is too large for every agent call. Create a ~200-line condensed version focusing on validation rules. The full doc remains in `docs/` for human reference and RAG ingestion.

5. **Scoring thresholds** — total deductions sum to ~2.0+ if everything fails, but realistically most emails will hit 0.3-0.6 in deductions. `passed` threshold remains 0.5 (score >= 0.5 passes), consistent with other checks.
