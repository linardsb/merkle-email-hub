# Plan: 49.3 Section-to-Component Classification Improvements

## Context

The converter's component matcher picks the wrong component for ~60% of sections on real Figma designs. Four systematic confusions occur across designs generically:

1. **editorial-2 false positive**: Any single-column section with 1 image + text triggers `editorial-2` (0.92) because `len(col_groups) == 1` fires even when the col_group represents a single stacked column, not a side-by-side editorial layout.
2. **article-card beats col-icon**: Small icon (≤60px) + text falls to `article-card` (0.9) because no icon-size check exists. `col-icon` isn't in the scorer at all.
3. **image-block beats full-width-image**: Image-only sections get `image-block` (1.0) regardless of width. `full-width-image` only triggers for `HERO` type sections (line 208), never for `CONTENT`.
4. **event-card too restrictive**: Requires `_DATE_PATTERN` regex + `has_images`, but structured event data (labeled "Date:", "Time:", "Location:" lines) without images misses entirely.

All fixes are **structural heuristics on `EmailSection` data** — they work on any Figma design, not design-specific patterns.

### EmailSection fields available for heuristics

`section_type`, `column_layout` (SINGLE/TWO/THREE/MULTI), `column_groups: list[ColumnGroup]`, `texts: list[TextBlock]` (.content, .is_heading, .font_size), `images: list[ImagePlaceholder]` (.width, .height, .node_id), `buttons: list[ButtonElement]`, `width`, `height`, `content_roles: tuple[str, ...]` (currently empty — populated only for MJML).

### Col_groups detection pipeline

1. **MJML names** → mj-column children → ColumnGroup (high reliability)
2. **Auto-layout HORIZONTAL** → frame children >40px → ColumnGroup (medium — can misfire on stacked content inside a horizontal parent)
3. **Y-position grouping** → ±10px bands → ColumnGroup (low — only fires when ≥2 children share Y)

**Root cause of editorial-2 false positive**: Strategy 2 creates a single ColumnGroup from a HORIZONTAL auto-layout parent that has one wide child frame containing both image and text. This is a **stacked layout**, not side-by-side columns, but it produces `col_groups=[ColumnGroup(images=[...], texts=[...])]` which triggers line 306.

## Implementation Steps

### Step 1: Fix editorial-2 false positive (`component_matcher.py:305-307`)

**Problem**: `len(col_groups) == 1 and col_groups[0].images and col_groups[0].texts` fires for any single col_group with mixed content — even a stacked single-column layout.

**Fix**: Add width validation. A real editorial column should be narrow (side-by-side with another). A single col_group that spans most of the container width is stacked content, not an editorial column.

```python
# component_matcher.py:305-307 — replace:
# editorial-2: single column group with mixed image+text (editorial layout)
if len(col_groups) == 1 and col_groups[0].images and col_groups[0].texts:
    candidates.append(("editorial-2", 0.92))

# with:
# editorial-2: needs GENUINE two-column structure, not just one col_group
# with mixed content. Two signals: (a) ≥2 col_groups each contributing content,
# or (b) 1 col_group that is narrow enough to be a real column (<70% of section width).
# When width is unknown (None), preserve backward compat — allow editorial-2.
if len(col_groups) >= 2:
    groups_with_content = sum(1 for g in col_groups if (g.images or g.texts))
    if groups_with_content >= 2:
        candidates.append(("editorial-2", 0.92))
elif len(col_groups) == 1 and col_groups[0].images and col_groups[0].texts:
    cg = col_groups[0]
    section_w = section.width or 600
    # Width unknown → keep editorial-2 (backward compat); width known → must be narrow
    cg_is_narrow = cg.width is None or cg.width < section_w * 0.7
    if cg_is_narrow:
        candidates.append(("editorial-2", 0.92))
    # else: wide single col_group → stacked layout, fall through to article-card
```

**Generic principle**: A real editorial layout has at least two visual columns or one narrow column that implies an off-screen partner. Width ratio is design-agnostic.

### Step 2: Add col-icon detector (`component_matcher.py:_score_extended_candidates`)

**Problem**: Small icon + text sections fall to `article-card` (0.9) because no icon-size heuristic exists.

**Fix**: Add `col-icon` detection in `_score_extended_candidates()` — icon-sized images (≤64px both dimensions) paired with 1-3 texts.

```python
# Add after the zigzag-alternating block (~line 432), before the fallback:

# col-icon: small icon image + short text group (icon-driven content block)
# Detects sections where imagery is decorative/icon-sized, not the primary content.
if len(images) == 1 and has_texts and 1 <= len(texts) <= 3:
    img = images[0]
    is_icon_sized = (
        img.width is not None and img.width <= 64
        and img.height is not None and img.height <= 64
    )
    if is_icon_sized:
        candidates.append(("col-icon", 0.92))
```

**Generic principle**: An image ≤64×64px is structurally an icon in any design. The 64px threshold covers common icon sizes (24, 32, 40, 48, 60, 64) while staying well below thumbnail/hero imagery.

### Step 3: Fix image-block vs full-width-image for CONTENT sections (`component_matcher.py:320-322`)

**Problem**: `image-block` (1.0) triggers for any `img==1 AND not has_texts`, even when the image spans full container width. `full-width-image` only appears for `HERO` type (line 208), never `CONTENT`.

**Fix**: Split the image-only path based on image width relative to container.

```python
# component_matcher.py:320-322 — replace:
# image-block: single image, no text
if img_count == 1 and not has_texts:
    candidates.append(("image-block", 1.0))

# with:
# full-width-image vs image-block: differentiate by image width relative to section
if img_count == 1 and not has_texts:
    img = section.images[0]
    section_w = section.width or 600
    is_full_width = (
        img.width is not None and img.width >= section_w * 0.8
    )
    if is_full_width:
        candidates.append(("full-width-image", 1.0))
    else:
        candidates.append(("image-block", 0.85))
    # Also add image-block as fallback when full-width wins, in case
    # slot fills differ downstream
    if not is_full_width:
        pass  # image-block already added above
    elif not has_texts:
        # full-width-image doesn't need caption; image-block does
        candidates.append(("image-block", 0.7))
```

**Simplified version** (cleaner):
```python
if img_count == 1 and not has_texts:
    img = section.images[0]
    section_w = section.width or 600
    if img.width is not None and img.width >= section_w * 0.8:
        candidates.append(("full-width-image", 1.0))
    else:
        candidates.append(("image-block", 1.0))
```

**Generic principle**: An image spanning ≥80% of its container is a full-width image in any email design. Smaller images (thumbnails, insets) are image-blocks.

### Step 4: Improve event-card detection (`component_matcher.py:417-419`)

**Problem**: Current event-card requires `has_images AND has_texts AND _DATE_PATTERN.search(all_text)`. Many event detail sections have no images and use keyword-labeled lines ("Date:", "Time:", "Location:") without matching the strict date regex.

**Fix**: Add keyword-based event detection alongside the existing date regex path.

```python
# New regex at module level (~line 353):
_EVENT_KEYWORD_PATTERN = re.compile(
    r"\b(?:date|time|location|venue|where|when|address|doors?\s+open)\s*:",
    re.IGNORECASE,
)
_TIME_OF_DAY_PATTERN = re.compile(
    r"\b\d{1,2}(?::\d{2})?\s*(?:am|pm|AM|PM)\b"
)

# component_matcher.py:417-419 — replace:
# event-card: date pattern in text, images, texts
if has_images and has_texts and _DATE_PATTERN.search(all_text):
    candidates.append(("event-card", 0.85))

# with:
# event-card: structured event information (date/time/location patterns)
# Path A: explicit date pattern (original) — works with or without images
if has_texts and _DATE_PATTERN.search(all_text):
    candidates.append(("event-card", 0.85))
# Path B: keyword-labeled event details (3+ short lines with event keywords)
elif has_texts and len(texts) >= 3:
    short_texts = [t for t in texts if len(t.content) < 80]
    if len(short_texts) >= 3:
        keyword_hits = sum(
            1 for t in short_texts
            if _EVENT_KEYWORD_PATTERN.search(t.content)
            or _TIME_OF_DAY_PATTERN.search(t.content)
        )
        if keyword_hits >= 2:
            candidates.append(("event-card", 0.83))
```

**Generic principle**: Event detail sections across any design share structural patterns — short labeled lines with temporal/spatial keywords. The two-path approach catches both formatted dates and keyword-labeled data.

### Step 5: Populate `content_roles` in layout_analyzer (`layout_analyzer.py:_classify_by_content`)

**Problem**: `content_roles` is always empty for non-MJML designs (line 142 default). This wastes a useful signal channel.

**Fix**: Populate content_roles during `_classify_by_content()` based on structural analysis. These are **soft hints** — the matcher can use them but doesn't depend on them.

Find `_classify_by_content()` and add content_roles computation. Since `EmailSection` is frozen, the roles need to be set during section construction in `analyze_layout()` or passed through. The cleanest approach: compute roles in a new helper and set them when building the EmailSection in the analysis loop.

```python
# New helper in layout_analyzer.py:
def _compute_content_roles(
    texts: list[TextBlock],
    images: list[ImagePlaceholder],
    buttons: list[ButtonElement],
) -> tuple[str, ...]:
    """Derive content role hints from section content (design-agnostic)."""
    roles: list[str] = []
    has_heading = any(t.is_heading for t in texts)
    has_icon = any(
        img.width is not None and img.width <= 64
        and img.height is not None and img.height <= 64
        for img in images
    )
    if texts and not images and not buttons:
        roles.append("text-only")
    if has_icon and texts:
        roles.append("text-with-icon")
    if images and texts and len(images) >= 1:
        large_imgs = [i for i in images if i.width is not None and i.width > 200]
        if large_imgs:
            roles.append("editorial")
    # Event-like: multiple short labeled lines
    short_labeled = [t for t in texts if len(t.content) < 80 and ":" in t.content]
    if len(short_labeled) >= 2:
        roles.append("event-info")
    return tuple(roles)
```

Wire into section construction in `analyze_layout()` where EmailSection is built.
