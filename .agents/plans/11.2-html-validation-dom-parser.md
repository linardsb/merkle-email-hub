# Plan: 11.2 HTML Validation — DOM Parser Upgrade

## Context
The current `html_validation` check uses 3 naive string-contains checks (`"<!DOCTYPE" in html`). It misses malformed structure, missing `<head>`/`<body>`, invalid nesting, missing charset, and other structural issues that AI agents commonly produce. This is the **foundation check** — if HTML structure is broken, all other QA checks give unreliable results, and the blueprint self-correction loop can't give actionable feedback to fixers.

**Goal:** Replace string matching with `lxml.html` DOM parsing. Validate 20 structural rules covering the full email HTML element space — from basic document skeleton through email-specific table layout to modern progressive enhancement elements. Return specific, actionable issue descriptions so the Recovery Router can route to the right fixer.

**Authoritative Reference:** `docs/html-email-components.md` — 280+ email-specific components/techniques/elements across 27 categories. Every Phase 11 check maps back to this doc.

## Build Profiles — Agency Methodology Adaptation

Different agencies build HTML email with fundamentally different methodologies. The QA system must adapt via **build profiles** — presets that bundle check configurations per methodology.

### Three Profiles

| Profile | Target Audience | Layout Approach | Key Characteristics |
|---------|----------------|-----------------|---------------------|
| `table_only` | Legacy agencies, Outlook-first B2B | `<table>` everywhere | No `<div>` layout, no progressive enhancement, strict table hierarchy |
| `hybrid` (default) | Most agencies | `<div>` + MSO `<table>` fallback | Fluid-hybrid columns, MSO conditionals required, responsive media queries |
| `modern_css` | Progressive agencies, mobile-first B2C | CSS Grid/Flexbox + fallbacks | Progressive enhancement elements, CSS `@supports`, advanced dark mode |

## 20 Structural Checks

### Group A: Document Skeleton (checks 1-5)

| # | Check | Why (agent failure mode) | Severity | Deduction |
|---|-------|--------------------------|----------|-----------|
| 1 | **DOCTYPE present** (`<!DOCTYPE html>`) | Agents sometimes omit it entirely | error | 0.15 |
| 2 | **`<html>` wraps `<head>` + `<body>`** | Agents produce `<head>` or `<body>` outside `<html>`, or omit one entirely | error | 0.15 |
| 3 | **`<meta charset="utf-8">`** (or `http-equiv` equivalent) in `<head>` | Agents frequently forget charset, causing encoding issues in non-Latin scripts | error | 0.15 |
| 4 | **`<meta name="viewport">`** in `<head>` | Critical for mobile rendering; agents skip it ~40% of the time | warning | 0.10 |
| 5 | **`<title>` in `<head>`** (non-empty) | Gmail/webmail show blank tab titles; agents often omit | warning | 0.10 |

### Group B: Tag Integrity (checks 6-8)

| # | Check | Why (agent failure mode) | Severity | Deduction |
|---|-------|--------------------------|----------|-----------|
| 6 | **Unclosed block-level tags** (`<div>`, `<table>`, `<tr>`, `<td>`, `<th>`, `<p>`, `<section>`, `<header>`, `<footer>`) | Most common agent structural error — unclosed `<td>` or `<div>` breaks layout | error | 0.15 per tag |
| 7 | **Block-in-inline nesting** (`<div>` inside `<span>`, `<table>` inside `<a>`, `<p>` inside `<p>`, etc.) | Agents nest blocks inside inlines, causing client-dependent parse differences | warning | 0.10 per violation |
| 8 | **Duplicate `id` attributes** | Agents duplicate IDs when generating repeated sections; breaks anchor links and tracking | warning | 0.10 per duplicate |

### Group C: Content Integrity (checks 9-10)

| # | Check | Why (agent failure mode) | Severity | Deduction |
|---|-------|--------------------------|----------|-----------|
| 9 | **Empty `<head>` or `<body>`** | Agent generates structural skeleton but no actual content | error | 0.15 |
| 10 | **`<style>` placement** — must be inside `<head>`, not `<body>`; no external `<link rel="stylesheet">` | Agents put `<style>` in `<body>` (breaks in many email clients); external stylesheets not supported by any email client | warning | 0.10 per violation |

### Group D: Email-Specific Structure (checks 11-14)

| # | Check | Why (agent failure mode) | Severity | Deduction |
|---|-------|--------------------------|----------|-----------|
| 11 | **Table structure integrity** — `<td>`/`<th>` must be inside `<tr>`, `<tr>` must be inside `<table>`/`<thead>`/`<tbody>`/`<tfoot>`, no text nodes directly inside `<table>` or `<tr>`, no invalid direct children of `<table>` | Email layout is 100% table-based. Agents produce orphaned `<td>` without `<tr>`, bare text inside `<table>`, `<tr>` outside `<table>`. These render unpredictably across every client. | error | 0.15 per violation |
| 12 | **List structure integrity** — `<li>` must be inside `<ul>` or `<ol>`. No orphaned `<li>` elements. `<ul>`/`<ol>` should only contain `<li>` as direct children (no bare text or `<div>` directly inside). | Agents generate bullet lists for features/benefits. Orphaned `<li>` outside `<ul>`/`<ol>` renders as unstyled text in some clients and vanishes in others. Lists ARE widely supported (Gmail, Apple Mail, Yahoo, Outlook). | warning | 0.10 per violation |
| 13 | **Duplicate structural elements** — multiple `<head>`, `<body>`, or `<title>` tags | Agents occasionally produce two `<body>` sections or duplicate `<head>` wrappers when combining generated blocks | error | 0.15 per duplicate |
| 14 | **Nested `<a>` tags** — `<a>` inside `<a>` | Illegal per HTML spec. Agents generate nested links when wrapping image-link patterns. Browsers split them unpredictably; email clients vary wildly. | error | 0.15 per violation |

### Group E: Progressive Enhancement & Modern Email Elements (checks 15-20)

These elements are **legitimate in modern email** and must NOT be flagged as forbidden. The DOM check validates they're *structurally correct* when present.

| # | Check | What it validates | Supported by | Severity | Deduction |
|---|-------|-------------------|-------------|----------|-----------|
| 15 | **`<video>` structure** | Must have `poster` attribute (static fallback image) AND fallback content between tags (for non-supporting clients). `<source>` children should have `type` attribute. | Apple Mail, iOS Mail, Thunderbird | warning | 0.10 per violation |
| 16 | **`<audio>` structure** | Same pattern as `<video>`: must have fallback content between tags for non-supporting clients. `<source>` children should have `type` attribute. Used in podcast/music email templates. | Apple Mail, Thunderbird | warning | 0.10 per violation |
| 17 | **`<picture>` + `<source>` structure** | Must contain exactly one `<img>` as fallback (last child). `<source>` children should have `srcset` and either `type` or `media` attribute. | Apple Mail, iOS | warning | 0.10 per violation |
| 18 | **Interactive & structured data elements** | **`<details>`** must have `<summary>` as first child (CSS-only accordions). **`<input type="checkbox/radio">`** must have matching `<label for="id">` (kinetic email pattern). **`<script type="application/ld+json">`** must contain valid JSON (Gmail structured data for action buttons, order tracking, event cards). **Inline `<svg>`** must have `xmlns` attribute and `role="img"` + `aria-label` for accessibility. | Apple Mail, iOS (details/input); Gmail (ld+json); Apple Mail (SVG) | warning | 0.10 per violation |
| 19 | **`<template>` element present** | The HTML `<template>` element makes its content **inert** — not rendered by browsers or email clients. Unlike stripped elements where clients remove the tag but may show content, `<template>` actively suppresses rendering by spec. If an agent produces this (from web component patterns), content is silently hidden. Similar to `<base>` — doesn't get stripped, actively causes damage. | N/A — always harmful in email | error | 0.15 |
| 20 | **`<base>` tag present** | Agents occasionally generate `<base href="...">` from web templates. In email, this silently breaks ALL relative URLs, tracking links, and image paths. Unlike other elements, `<base>` is NOT stripped — it actively causes damage. | N/A — always harmful in email | error | 0.15 |

**Scoring:** Start at 1.0, deduct per issue (values above). Minimum 0.0. `passed = score >= threshold` (default threshold 0.5 from config).

## Dynamic Approach — Not Overwhelming AI Agents

The 280+ elements from `docs/html-email-components.md` are the **complete inventory** — but agents never see all 280 at once. The system uses progressive filtering to keep agent cognitive load minimal:

### Layer 1: Build Profile Pre-Filtering
The build profile (`table_only` / `hybrid` / `modern_css`) **skips entire check groups** before any agent runs. A `table_only` project skips progressive enhancement (video, audio, picture, interactive), dark mode, and advanced CSS checks. Fewer checks = fewer possible failures = less noise.

### Layer 2: Independent Domain Checks (10 checks, each focused)
Each QA check owns one domain and reports only its own issues:
- `html_validation` reports structural DOM issues — never mentions dark mode or CSS support
- `dark_mode` reports color-scheme issues — never mentions table hierarchy
- `css_support` reports unsupported properties — never mentions accessibility

An agent fixing a structural issue doesn't see dark mode failures. They're separate results in separate domains.

### Layer 3: Recovery Router Serializes Fixes
When multiple checks fail, the Recovery Router picks the **highest priority failure domain** and routes to ONE specialist agent:
1. Structure failures → Scaffolder (rebuild skeleton)
2. MSO failures → Outlook Fixer
3. Dark mode failures → Dark Mode agent
4. Accessibility failures → Accessibility Auditor
5. Content failures → Content agent

The agent receives **3-8 specific actionable issues** per iteration, not a dump of all failures. Example: `"Unclosed <td> at nesting depth 3"`, `"<video> missing poster attribute"`, `"Duplicate id='hero-cta'"` — not "check all 280 elements."

### Layer 4: Bounded Self-Correction
The blueprint engine caps retries (default 3 per agent, 5 per pipeline). If an agent can't fix its domain's failures in 3 attempts, it escalates to the Recovery Router which may try a different agent or halt with partial results. No infinite loops through the element inventory.

### Layer 5: Actionable Message Format
Every QA issue includes:
- **What's wrong**: specific element/location (e.g., `"<td> outside <tr> in table at line ~42"`)
- **Why it matters**: rendering consequence (e.g., `"renders unpredictably across clients"`)
- **How to fix**: concrete instruction (e.g., `"wrap <td> in <tr>"`)

This gives the fixer agent a clear, bounded task — not "validate against 280 elements."

### Net Effect
```
280+ elements (complete inventory)
  → Build profile filters    → ~180-220 relevant elements
  → 10 independent checks    → each check sees ~15-30 elements in its domain
  → Agent sees failures only → 3-8 specific issues per self-correction iteration
  → Bounded retries          → max 3 attempts per agent, then escalate
```

### Elements Intentionally NOT Checked (no validation needed)

| Element | Why no check |
|---------|-------------|
| `<br>`, `<hr>`, `<wbr>` | Self-closing, no structure to validate. Universally supported. |
| `<center>`, `<font>` | Deprecated in HTML5 but **intentionally used in email** for Outlook compatibility. Must NOT be flagged. |
| `<strong>`, `<em>`, `<b>`, `<i>`, `<u>`, `<s>`, `<small>`, `<mark>`, `<del>`, `<ins>` | Inline text formatting. Covered by nesting rules (check 7). Universally supported. |
| `<sup>`, `<sub>` | Superscript/subscript (trademarks, footnotes). No structural requirements. Widely supported. |
| `<abbr>`, `<cite>`, `<q>`, `<time>`, `<data>`, `<code>` | Inline semantic elements. No structural validation needed. |
| `<article>`, `<aside>`, `<nav>`, `<main>`, `<figure>`, `<figcaption>` | HTML5 semantic containers. Treated as `<div>` by email clients. Covered by unclosed tag tracking (check 6). |
| `<noscript>` | Valid in email, shows content when JS disabled (always in email). No structural concern. |
| `<map>`, `<area>` | Image maps. Extremely rare in modern email. `<area href>` covered by 11.7 if present. |
| `<iframe>`, `<embed>`, `<object>`, `<canvas>`, `<applet>` | Stripped by all/most email clients. No rendering damage — content just disappears. Not worth flagging. |
| `<form>`, `<select>`, `<textarea>`, `<button>` | Very limited email support. If used with `<input>` + `<label>` for kinetic email, covered by check 18. Standalone form elements too niche to validate. |
| `<meter>`, `<progress>` | Niche gamification elements. Too rare in agent output. |
| `<ruby>`, `<rt>`, `<rp>` | CJK text annotations. Extremely niche. |
| `<dialog>` | Not used in email. |
| HTML comments `<!-- -->` | Regular comments add to file size (covered by 11.8). MSO conditional comments covered by 11.4. |
| `bgcolor`, `cellpadding`, `cellspacing`, `border`, `valign`, `align`, `width` (attributes) | Outlook rendering attributes. These are HTML attributes, not elements. Best practices, not structural validation. |

- Table structure: validate parent/child relationships using defined valid-children and required-parents maps

- Progressive elements: only check structure when element IS present (no false positives for absence)
- `<script type="application/ld+json">`: validate JSON parse, distinguish from executable `<script>` (which is harmless — just stripped)
- Inline `<svg>`: check for `xmlns="http://www.w3.org/2000/svg"` and accessibility attributes
- `<input>` + `<label>`: collect all `<input>` with `id`, verify matching `<label for="id">` exists
- Duplicate structural detection: lxml may merge duplicate `<head>`/`<body>` during parsing — fall back to raw string counting if needed

## Technical Notes

- **lxml `document_fromstring`** tolerates malformed HTML (like browsers do). We use that tolerance to parse, then compare against raw HTML for unclosed tag detection.
- **Backward compatible**: `config` param stays optional (`None` = hardcoded defaults matching `defaults.yaml`). Same `QACheckResult` return type. Same `name = "html_validation"`.
- **Fixture update**: Adding meta tags and title to `sample_html_valid` may surface new failures in other check tests. The minimal fixture stays unchanged.
- **lxml auto-normalization caveat**: lxml may merge duplicate `<head>`/`<body>` during parsing. Test during implementation and fall back to raw string detection if needed.
