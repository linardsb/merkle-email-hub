### Step 2: SDK Regeneration

1. **Start backend** (needed for live OpenAPI spec):
   ```bash
   make dev &  # or just the backend
   ```

2. **Fetch fresh OpenAPI spec + regenerate SDK:**
   ```bash
   make sdk
   ```
   This runs `cd cms && pnpm --filter @email-hub/sdk generate-sdk:fetch` which:
   - Fetches `http://localhost:8891/openapi.json`
   - Writes to `cms/packages/sdk/openapi.json`
   - Runs `@hey-api/openapi-ts` to generate `types.gen.ts`, `sdk.gen.ts`, `client.gen.ts`

3. **If backend cannot be started** (e.g., missing DB), use the alternative:
   - Capture OpenAPI JSON manually: `curl http://localhost:8891/openapi.json > cms/packages/sdk/openapi.json`
   - Then: `make sdk-local`

4. **Verify SDK output** — check that `cms/packages/sdk/src/client/types.gen.ts` now contains:
   - `ConnectionResponse` / `ConnectionCreateRequest`
   - `FileStructureResponse` / `ComponentListResponse`
   - `LayoutAnalysisResponse` / `GenerateBriefResponse`
   - `ImportResponse` / `StartImportRequest` / `ConvertImportRequest`
   - `ExtractComponentsResponse` / `ExtractComponentsRequest`
   - All 17 design-sync endpoint functions in `sdk.gen.ts`

### Step 3: Frontend Type Migration Decision

The frontend currently uses **manual types** in `cms/apps/web/src/types/design-sync.ts` (186 lines) imported by `use-design-sync.ts` hooks. The SDK will generate equivalent types.

**Recommended approach**: Keep manual types for now. Migrating 8 hooks + multiple components to SDK types is a larger refactor that risks breaking the frontend. The manual types are well-maintained and match the API. A future task can migrate to SDK types.

**Minimal action**: Verify `make check-fe` passes after SDK regeneration (no type conflicts).
