# Plan: Phase 32.10 — Skill Versioning with Rollback

## Context

Phase 32.6 introduces automated skill file patching via the eval pipeline. Without versioning, there's no clean rollback path beyond `git revert`. Versioning adds: (a) version tracking per skill file, (b) runtime pinning to a prior version without touching disk, (c) A/B testing old vs new via the existing override mechanism.

## Research Summary

### Current Skill Loading Flow

```
prompt.py → get_override(agent) → SKILL.md on disk
           → extract_skill_for_mode(content, mode)
           → L3 skills loaded by detect_relevant_skills() per agent prompt.py
           → parse_skill_meta(content) → SkillMeta(token_cost, priority)
           → should_load_skill() budget check
```

### Current L3 Frontmatter

Inconsistent across agents:
- Some files: `---\npriority: 3\n---` (e.g., `dark_mode/skills/client_behavior.md`)
- Some files: HTML comment only `<!-- L4 source: ... -->` (e.g., `scaffolder/skills/table_layouts.md`)
- Some files: No frontmatter at all

`parse_skill_meta()` handles missing frontmatter gracefully (returns defaults).

### Skill Override Mechanism (`skill_override.py`)

Simple in-memory dict: `set_override(agent, content)` / `get_override(agent)` / `clear_override(agent)`.
Priority: prompt store DB > in-memory override > SKILL.md on disk.
Used by `skill_ab.py` for A/B testing (swap SKILL.md content during eval runs).
**Currently operates on SKILL.md (L1) only, NOT on L3 skill files.**

## Design Decisions

### D1: Version manifest location
**Decision:** One `skill-versions.yaml` per agent directory (e.g., `app/ai/agents/dark_mode/skill-versions.yaml`).
**Why:** Collocates with the skill files it tracks. Easy to review in PRs. Follows VSA pattern.

### D2: YAML parsing
**Decision:** Use `yaml` stdlib (PyYAML already a dependency via Pydantic/other tools). `skill-versions.yaml` is a structured config file, not frontmatter — regex parsing would be fragile.
**Why:** The manifest has nested dicts with typed values (floats, nulls, dates). Regex won't cut it.

### D3: Version pinning targets L3 skill files, not SKILL.md
**Decision:** Pinning operates on individual L3 skill files via `git show`. SKILL.md pinning uses the existing `skill_override.py` mechanism unchanged.
**Why:** L3 files are what 32.6 patches. SKILL.md already has the override system. No point duplicating.

### D4: Frontmatter version field
**Decision:** Add `version: "1.0.0"` to frontmatter. Files without it default to `"1.0.0"`. Semver format for consistency with the manifest.
**Why:** Lightweight. Backward compatible. `parse_skill_meta()` already handles missing fields.

### D5: `git show` for pinned content
**Decision:** Load pinned versions via `subprocess.run(["git", "show", f"{hash}:{path}"])`. Validate hash and path against the manifest before execution.
**Why:** No need to store full file content in the manifest. Git is the source of truth. Hash is validated (not user input).

## Implementation Steps

### Step 1: Create `SkillVersionEntry` schema in `evals/schemas.py`

```python
@dataclass(frozen=True)
class SkillVersionEntry:
    """A single version record in the skill-versions.yaml manifest."""
    version: str  # Semver: "1.0.0"
    hash: str  # Git short hash
    date: str  # ISO date: "2026-03-27"
    source: str = "manual"  # "manual" | "eval-driven"
    eval_pass_rate: float | None = None
```

### Step 2: Create `app/ai/agents/skill_version.py`

Core module with:

```python
@dataclass(frozen=True)
class SkillVersionManifest:
    """Parsed skill-versions.yaml for one agent."""
    agent: str
    skills: dict[str, SkillVersionConfig]  # skill_name → config

@dataclass(frozen=True)
class SkillVersionConfig:
    """Version config for a single skill file."""
    current: str  # Current version on disk
    pinned: str | None  # Pinned version (None = use current)
    versions: dict[str, SkillVersionEntry]  # version → entry
```

Functions:
- `load_manifest(agent: str) -> SkillVersionManifest | None` — parse YAML from `app/ai/agents/{agent}/skill-versions.yaml`, return None if missing
- `save_manifest(manifest: SkillVersionManifest) -> None` — write YAML back
- `pin_skill(agent: str, skill_name: str, version: str) -> None` — validate version exists, set `pinned` in manifest
- `unpin_skill(agent: str, skill_name: str) -> None` — set `pinned: null`
- `list_skill_versions(agent: str, skill_name: str) -> list[SkillVersionEntry]` — return version history sorted by date desc
- `load_pinned_content(agent: str, skill_name: str) -> str | None` — if pinned, run `git show {hash}:{path}`, return content. If not pinned, return None (caller uses disk file)
- `record_version(agent: str, skill_name: str, version: str, git_hash: str, source: str, eval_pass_rate: float | None) -> None` — add version entry to manifest, update `current`
- `bump_version(current: str) -> str` — semver minor bump: "1.0.0" → "1.1.0"
- `_resolve_skill_path(agent: str, skill_name: str) -> str` — map skill_name to relative file path using `CRITERION_SKILL_MAP` or convention

Security: `_VALID_HASH_RE = re.compile(r"^[0-9a-f]{7,40}$")` to validate git hashes. Path validated against manifest (no user-supplied paths to `git show`).

### Step 4: Integrate pinning into agent prompt loading

Each agent's `prompt.py` calls `_load_skill_file(name)` which reads from disk. Modify the L3 skill loading pattern:

In `skill_loader.py`, add a helper:
```python
def load_skill_with_version(agent: str, skill_name: str, skill_path: Path) -> tuple[str, str]:
    """Load skill content, respecting version pins.

    Returns (content, loaded_version).
    """
    from app.ai.agents.skill_version import load_pinned_content, load_manifest

    pinned = load_pinned_content(agent, skill_name)
    if pinned is not None:
        meta, body = parse_skill_meta(pinned)
        return pinned, meta.version

    content = skill_path.read_text(encoding="utf-8")
    meta, body = parse_skill_meta(content)
    return content, meta.version
```

**Note:** Agent prompt.py files currently load L3 skills directly via `_load_skill_file()`. The version check wraps this — no changes to individual prompt.py files needed if we hook into the loading path.

### Step 5: Add `set_override_from_version()` to `skill_override.py`

```python
def set_override_from_version(agent: str, skill_name: str, version: str) -> None:
    """Load skill content from a specific version and set as override.

    Used for A/B testing version X vs version Y without changing disk files.
    """
    from app.ai.agents.skill_version import load_manifest, _git_show_content, _resolve_skill_path

    manifest = load_manifest(agent)
    if manifest is None:
        raise ValueError(f"No skill-versions.yaml for agent: {agent}")

    config = manifest.skills.get(skill_name)
    if config is None:
        raise ValueError(f"Skill {skill_name!r} not in manifest for {agent}")

    entry = config.versions.get(version)
    if entry is None:
        raise ValueError(f"Version {version!r} not found for {agent}/{skill_name}")

    path = _resolve_skill_path(agent, skill_name)
    content = _git_show_content(entry.hash, path)
    set_override(agent, content)
```

### Step 6: Integrate with `skill_updater.py`

After `_apply_patches_git()` succeeds, update version manifests:

```python
def _update_version_manifests(patches: list[SkillFilePatch], git_hash: str) -> None:
    """Bump version in frontmatter and update skill-versions.yaml for each patched file."""
    from app.ai.agents.skill_version import load_manifest, record_version, bump_version
    from app.ai.agents.skill_loader import parse_skill_meta

    for patch in patches:
        path = Path(patch.skill_file_path)
        if not path.exists():
            continue

        content = path.read_text()
        meta, body = parse_skill_meta(content)
        new_version = bump_version(meta.version)

        # Update frontmatter in file
        _update_frontmatter_version(path, content, new_version)

        # Update manifest
        agent = patch.candidate.agent
        skill_name = path.stem  # e.g., "client_behavior"
        record_version(
            agent=agent,
            skill_name=skill_name,
            version=new_version,
            git_hash=git_hash,
            source="eval-driven",
            eval_pass_rate=patch.candidate.pass_rate,
        )
```

Add call in `_apply_patches_git()` after the git commit, using `git rev-parse --short HEAD` for the hash.

## Preflight Warnings

- `skill_override.py` currently only overrides SKILL.md (L1), not L3 files. `set_override_from_version()` for L3 files needs a separate dict (not `_overrides`), or the pinning should be handled at the `load_skill_with_version()` level (bypassing override). **Recommendation:** Keep L3 version pinning in `skill_version.py` (via `load_pinned_content`), only use `set_override_from_version()` for SKILL.md A/B comparisons.
- `parse_skill_meta()` uses regex, not YAML. The `version:` field must be parsed as a string (strip quotes). Ensure `"1.0.0"` and `1.0.0` both work.
- L3 skill files are loaded by individual `prompt.py` files, each with their own `_load_skill_file()`. The version hook must intercept at a common point. Best approach: add `load_skill_with_version()` to `skill_loader.py` and have prompt.py files call it.
- `import_annotator` has L3 files in `skills/l3/` subdirectory — manifest must handle this nested path.

## Security Checklist

- **No new API endpoints** — all operations are CLI/internal only
- **git show injection:** Hash validated via `_VALID_HASH_RE = re.compile(r"^[0-9a-f]{7,40}$")`, path validated against manifest
- **skill-versions.yaml** checked into repo — changes reviewed via PR
- **No user input reaches subprocess** — all values from manifest (trusted)
- Pinning does not modify skill files on disk — runtime only
