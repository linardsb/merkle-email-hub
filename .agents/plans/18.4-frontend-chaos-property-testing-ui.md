# Plan: 18.4 Frontend Chaos & Property Testing UI

## Context
Phase 18.1 (Chaos Engine) and 18.2 (Property-Based Testing) added backend endpoints for resilience and invariant testing. Developers need a UI to run these tests, visualize results, and act on failures. This plan adds two panels to the QA sidebar, SWR hooks, SDK types, and i18n keys.

## Backend Endpoints (already exist)
- `POST /api/v1/qa/chaos-test` (3/min) → `ChaosTestResponse`
- `POST /api/v1/qa/property-test` (1/min) → `PropertyTestResponse`

## Response Schemas (from `app/qa_engine/schemas.py`)

**ChaosTestResponse:**
```
original_score: float (0-1)
resilience_score: float (0-1)
profiles_tested: int
profile_results: list[ChaosProfileResult]  # per-profile breakdown
critical_failures: list[ChaosFailure]
```
**ChaosProfileResult:** `profile, description, score, passed, checks_passed, checks_total, failures: list[ChaosFailure]`
**ChaosFailure:** `profile, check_name, severity, description`

**PropertyTestResponse:**
```
total_cases: int
passed: int
failed: int
failures: list[PropertyFailureSchema]
seed: int
invariants_tested: list[str]
```
**PropertyFailureSchema:** `invariant_name, violations: list[str], config: dict`
**PropertyTestRequest:** `invariants: list[str] | None, num_cases: int (1-1000, default 100), seed: int | None`

## Files to Create

### 1. `cms/apps/web/src/types/chaos.ts` — Frontend type aliases
Re-export SDK types + define local interfaces for chaos/property testing.

### 2. `cms/apps/web/src/hooks/use-chaos-test.ts` — SWR hook
- `useChaosTest()` — `useSWRMutation` calling `POST /api/v1/qa/chaos-test` via `longMutationFetcher` (120s timeout — chaos runs QA checks per profile)
- Request type: `{ html: string; profiles?: string[] }`
- Returns `{ trigger, data, error, isMutating }`

### 3. `cms/apps/web/src/hooks/use-property-test.ts` — SWR hook
- `usePropertyTest()` — `useSWRMutation` calling `POST /api/v1/qa/property-test` via `longMutationFetcher` (generates up to 1000 emails)
- Request type: `{ invariants?: string[]; num_cases?: number; seed?: number }`
- Returns `{ trigger, data, error, isMutating }`

### 4. `cms/apps/web/src/components/qa/ChaosTestPanel.tsx`
Compact panel widget (like `VisualQAPanelTab`) embedded in `QAResultsPanel`.

**Layout:**
- Header: "Chaos Testing" label + "Run" button (disabled while mutating)
- When no results: placeholder text
- When results:
  - **Score bar:** `resilience_score` as colored progress bar (green ≥0.8, yellow ≥0.5, red <0.5) with `original_score` comparison
  - **Profile results grid:** Each profile as a collapsible row:
    - Profile name + description + score badge (pass/fail colored)
    - `checks_passed/checks_total` progress
    - Expanded: failure list with severity badges (error=red, warning=yellow, info=muted)
  - **Critical failures section** (if any): highlighted at top with red bg

**"Fix This" action:** On each failure row, a "Fix" button that triggers `useQARun` with the original HTML — this dispatches to the existing QA pipeline which includes the CRAG validation loop and repair pipeline. The fix action is available when `html` prop is provided.

**Props:** `html: string` (required — the email HTML to test)

### 5. `cms/apps/web/src/components/qa/PropertyTestPanel.tsx`
Compact panel widget embedded in `QAResultsPanel`.

**Layout:**
- Header: "Property Testing" label + "Run" button with configurable case count (dropdown: 50/100/500)
- When no results: placeholder text
- When results:
  - **Pass/fail gauge:** `passed/total_cases` as progress bar + percentage
  - **Seed display:** Small muted text showing seed for reproducibility
  - **Invariants tested:** Chip list of invariant names (green=all passed, red=has failures)
  - **Failures list:** Each failure as collapsible row:
    - Invariant name + violation count badge
    - Expanded: violation strings as list items
    - Expanded: "Config" toggle showing minimized `EmailConfig` as formatted key-value pairs (not raw JSON dump — pick the ~6 most relevant fields based on invariant type)

**Props:** none required (property testing generates its own emails)

## Implementation Steps

### Step 1: SDK Types
1. Run `make sdk` to regenerate SDK types from OpenAPI spec
2. Create `cms/apps/web/src/types/chaos.ts`:
   - Import and re-export SDK types (or define local interfaces if SDK gen didn't pick them up)
   - Types needed: `ChaosTestRequest`, `ChaosTestResponse`, `ChaosProfileResult`, `ChaosFailure`, `PropertyTestRequest`, `PropertyTestResponse`, `PropertyFailureSchema`

### Step 2: SWR Hooks
3. Create `cms/apps/web/src/hooks/use-chaos-test.ts`:
   ```typescript
   import useSWRMutation from "swr/mutation";
   import { longMutationFetcher } from "@/lib/fetchers";
   import type { ChaosTestRequest, ChaosTestResponse } from "@/types/chaos";
   import type { ApiError } from "@/types/api";

   export function useChaosTest() {
     return useSWRMutation<ChaosTestResponse, ApiError, string, ChaosTestRequest>(
       "/api/v1/qa/chaos-test",
       longMutationFetcher
     );
   }
   ```

4. Create `cms/apps/web/src/hooks/use-property-test.ts`:
   ```typescript
   import useSWRMutation from "swr/mutation";
   import { longMutationFetcher } from "@/lib/fetchers";
   import type { PropertyTestRequest, PropertyTestResponse } from "@/types/chaos";
   import type { ApiError } from "@/types/api";

   export function usePropertyTest() {
     return useSWRMutation<PropertyTestResponse, ApiError, string, PropertyTestRequest>(
       "/api/v1/qa/property-test",
       longMutationFetcher
     );
   }
   ```

## Component Patterns to Follow

Reference these existing components for consistency:
- **Panel widget pattern:** `cms/apps/web/src/components/visual-qa/visual-qa-panel-tab.tsx` — compact panel with action button + results display
- **SWR mutation pattern:** `cms/apps/web/src/hooks/use-visual-qa.ts` — `useSWRMutation` with `longMutationFetcher`
- **Type re-export pattern:** `cms/apps/web/src/types/rendering.ts` — SDK re-exports + local types
- **Collapsible sections:** `cms/apps/web/src/components/workspace/qa-results-panel.tsx` — `useState` toggle with chevron icon
- **Badge/severity pattern:** existing QA check items in `qa-check-item.tsx`
- **i18n pattern:** `useTranslations("namespace")` with ICU message format placeholders

## Tailwind Token Rules
- Use semantic tokens only: `text-foreground`, `text-muted-foreground`, `bg-background`, `bg-muted`, `bg-card`, `border-border`, `text-destructive`, `bg-destructive`
- NEVER use primitive colors (`text-gray-500`, `bg-blue-600`)
- Score thresholds: `bg-success` (≥0.8), `bg-warning` (≥0.5), `bg-destructive` (<0.5)

## Security Checklist
- [x] No raw HTML rendered in UI — all chaos/property results are structured data (scores, strings, numbers)
- [x] "Fix This" action uses existing `useQARun` hook which calls authenticated endpoint

- [x] Config dict displayed as text key-value pairs, not interpolated into DOM
