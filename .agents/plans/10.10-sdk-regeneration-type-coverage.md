# Plan: 10.10 SDK Regeneration & Type Coverage

## Context

The OpenAPI TypeScript SDK (`cms/packages/sdk/`) is stale. The `openapi.json` snapshot was generated early in the project and only covers Phase 0-2 endpoints. Phase 3-10 backend routes (templates, rendering, memory, blueprints, ontology, graph search, failure patterns, agents) are registered in `app/main.py` but missing from the SDK. Consequently:

1. **`sdk.gen.ts`** has ~40 endpoint functions but is missing ~30+ endpoints for templates, rendering, memory, blueprints, ontology, graph search, failure patterns, and agent routes
2. **`types.gen.ts`** has *some* Phase 8-10 types (e.g., `BlueprintRunResponse`, `GraphSearchResult`, `EmailClientResponse`) but is missing others (e.g., `FailurePatternResponse`, `MemoryResponse`, rendering types)
3. **18 local type files** in `cms/apps/web/src/types/` duplicate backend schemas because hooks couldn't import from SDK — these should be replaced with SDK imports after regeneration

### Step 2: Audit generated types against local types

For each local type file, compare field-by-field with the SDK-generated equivalent. The SDK types are authoritative (generated from Pydantic schemas). If a local type has extra fields not in the backend schema, that's a frontend-only extension that should stay local but re-export SDK base types.

Pattern for migration:

```typescript
// BEFORE (types/templates.ts):
export interface TemplateResponse {
  id: number;
  project_id: number;
  name: string;
  // ... manually maintained
}

// AFTER (types/templates.ts):
export type { TemplateResponse, TemplateCreate, TemplateUpdate, VersionResponse } from "@email-hub/sdk";

// Keep only frontend-specific types that have no backend equivalent:
export interface PaginatedTemplates {
  items: TemplateResponse[];
  total: number;
  page: number;
  page_size: number;
}
```

**Note:** If the SDK generates a paginated response type (e.g., `PaginatedResponseTemplateResponse`), use that instead of the local `PaginatedTemplates`.

### Step 3: Migrate each local type file

Apply the migration pattern from Step 2 to each file. Order of migration (by dependency — migrate leaf types first):

1. **`types/templates.ts`** — Remove local interfaces, re-export from SDK. Keep `PaginatedTemplates` only if SDK doesn't have equivalent.
2. **`types/rendering.ts`** — Re-export `RenderingTest`, `ScreenshotResult`, `RenderingComparisonResponse`, etc. from SDK.
3. **`types/failure-patterns.ts`** — Re-export `FailurePatternResponse`, `FailurePatternListResponse`, `FailurePatternStats` from SDK.
4. **`types/graph-search.ts`** — Re-export `GraphSearchResponse`, `GraphSearchResult`, `GraphEntity`, `GraphRelationship` from SDK. Keep `SearchMode` and `GraphSearchArgs` if not in SDK (these are frontend-only UI types).
5. **`types/knowledge.ts`** — Re-export knowledge document types from SDK. Keep any frontend-only extensions.
6. **`types/qa.ts`** — Re-export QA types from SDK.
7. **`types/connectors.ts`** — Check overlap with existing SDK `ExportRequest`/`ExportResponse` imports. Re-export additional types.
8. **`types/blueprint-runs.ts`** — Already imports from SDK. `BlueprintRunRecord` is a frontend wrapper — keep it local. Verify `BlueprintRunResponse` still matches.
9. **`types/chat.ts`** and **`types/chat-history.ts`** — Chat/streaming types may not have direct backend Pydantic schemas (SSE streaming). Keep local if not in SDK.

### Step 4: Update hook imports

For each hook that was importing from `@/types/*`, update the import path:

```typescript
// BEFORE:
import type { FailurePatternListResponse, FailurePatternStats } from "@/types/failure-patterns";

// AFTER (if types/failure-patterns.ts now re-exports from SDK):
// No change needed — the re-export is transparent

// OR (direct SDK import, preferred for types that are 1:1 with SDK):
import type { FailurePatternListResponse, FailurePatternStats } from "@email-hub/sdk";
```

**Decision:** Keep the `@/types/*` barrel re-exports as the canonical import path for hooks and components. This provides a stable import surface — if SDK type names change during regeneration, only the barrel files need updating.
